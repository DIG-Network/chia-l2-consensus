# API-008 — Return-Not-Submit Pattern

> **Authoritative requirement:** [API-008](../NORMATIVE.md#API-008)
> **Verification:** [VERIFICATION.md](../VERIFICATION.md)
> **Tracking:** [TRACKING.yaml](../TRACKING.yaml)
> **Spec reference:** [spec-consensus-crate.md](../../../../resources/spec-consensus-crate.md) — "The crate builds SpendBundles but NEVER broadcasts them"

## Summary

The crate is a library, not a service. Every public method that produces coin
spends returns a `SpendBundle` to the caller. The crate MUST NOT contain any
code that broadcasts, submits, or pushes transactions to a Chia full node.

The importing project (the L2 system) is solely responsible for:
- Choosing when to broadcast
- Selecting fees
- Retry logic on mempool rejection
- Combining bundles from multiple sources before broadcast
- Monitoring transaction inclusion

## Specification

### Methods That Return SpendBundle

| Method | What it builds | Caller responsibility |
|--------|---------------|----------------------|
| `deploy()` | Network coin + checkpoint singleton launchers | Broadcast to create the L2 network |
| `register_validator()` | Network coin spend + registration coin creation | Broadcast to register a validator |
| `build_checkpoint()` | Checkpoint singleton spend with Groth16 proof + BLS sig | Broadcast to advance epoch |
| `recover_collateral()` | Membership query + registration coin spend bundle | Broadcast to recover validator collateral |
| `query_membership_on_chain()` | Membership query singleton spend | Broadcast to emit on-chain announcement |

### What The Crate MUST NOT Do

The crate MUST NOT:
- Call `push_tx()`, `send_transaction()`, or any RPC that submits transactions
- Hold a reference to a full node client for the purpose of broadcasting
- Retry failed broadcasts internally
- Set transaction fees (the SpendBundle has no fee; the caller wraps or adds fee coins)
- Block waiting for transaction confirmation

### What The Crate MAY Do

The crate MAY:
- Accept a `FullNodeClient` (or equivalent) for **reading** chain state (`sync()`)
- Query coin records, puzzle reveals, and blockchain state
- Use RPC for read-only operations during indexing

### Why This Design

1. **Separation of concerns**: The crate handles cryptography and puzzle construction;
   the L2 handles network operations and transaction lifecycle.

2. **Composability**: The L2 may combine consensus bundles with its own coin spends
   (e.g., adding fee coins, batching operations) before broadcasting.

3. **Testability**: Unit and integration tests can verify bundle construction without
   needing a live node or simulating network conditions.

4. **Control**: The L2 controls broadcast timing, which matters for checkpoint
   coordination (only one checkpoint in-flight at a time).

### Example Usage Pattern

```rust
// The L2 builds the bundle via the crate
let bundle = client.build_checkpoint(
    new_state_root,
    new_validator_merkle_root,
    new_validator_count,
    &signing_pubkeys,
    &signatures,
).await?;

// The L2 broadcasts the bundle itself
let inclusion = node.push_tx(bundle).await?;
match inclusion {
    Ok(status) => log::info!("Checkpoint accepted: {:?}", status),
    Err(e) => {
        // L2 decides retry strategy
        log::warn!("Checkpoint rejected: {:?}", e);
        // Maybe re-sync and retry, or alert operator
    }
}
```

### Previous Naming

The `build_checkpoint()` method was previously named `submit_checkpoint()` in
earlier spec drafts. The name was changed to `build_checkpoint()` to eliminate
any implication that the crate performs transaction submission.

## Acceptance Criteria

- [ ] `deploy()` returns `(SpendBundle, NetworkConfig)` — does not broadcast
- [ ] `register_validator()` returns `SpendBundle` — does not broadcast
- [ ] `build_checkpoint()` returns `SpendBundle` — does not broadcast
- [ ] `recover_collateral()` returns `SpendBundle` — does not broadcast
- [ ] `query_membership_on_chain()` returns `SpendBundle` — does not broadcast
- [ ] No `push_tx`, `send_transaction`, or equivalent exists anywhere in the crate
- [ ] `FullNodeClient` (if used) is only used for read-only RPC operations
- [ ] Integration test demonstrates caller-broadcasts pattern

## Verification

1. `grep -r "push_tx\|send_transaction\|broadcast" src/` returns zero matches
2. All five methods have return type containing `SpendBundle`
3. Integration test calls `sim.push_tx(bundle)` — the test, not the crate, broadcasts
4. No `FullNodeClient` method call in the crate writes to the blockchain
