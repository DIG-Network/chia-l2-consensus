# API-005 — ConsensusClient Message Computation

> **Authoritative requirement:** [API-005](../NORMATIVE.md#API-005)

> **Tracking:** [TRACKING.yaml](../TRACKING.yaml)

## Summary

`ConsensusClient` must provide facade methods for message computation, validator set transitions, and membership queries. These wrap the internal wire format and merkle functions with the client's current state.

## Specification

### checkpoint_message() (Spec Lines 1903-1918)

```rust
impl ConsensusClient {
    /// Compute the 32-byte checkpoint message for a proposed state transition.
    pub fn checkpoint_message(
        &self,
        new_state_root: Bytes32,
        new_validator_merkle_root: Bytes32,
        new_validator_count: u64,
    ) -> ConsensusResult<[u8; 32]>;
}
```

Uses current epoch+1 as new_epoch. Delegates to `compute_checkpoint_message()`.

### validator_signing_message() (Spec Lines 1920-1945)

## Acceptance Criteria

- [ ] `checkpoint_message()` matches `compute_checkpoint_message()` with current epoch+1
- [ ] `validator_signing_message()` is 96 bytes: msg + genesis + coin_id
- [ ] `compute_new_validator_set()` applies entries/exits and returns new root + count
- [ ] `is_active()` checks local merkle tree without RPC
- [ ] `membership_announcement()` uses current epoch and checkpoint coin ID
- [ ] All methods return `NotDeployed` before first `sync()`
