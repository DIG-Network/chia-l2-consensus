# NET-004 — Network Coin Self-Recreation

> **Authoritative requirement:** [NET-004](../NORMATIVE.md#NET-004)

> **Tracking:** [TRACKING.yaml](../TRACKING.yaml)

## Summary

After creating a registration coin, the network coin MUST recreate itself to remain available for future registrations. This is a fundamental singleton property: the network coin is never "consumed" — it persists through all registration events, maintaining a continuous lineage from deployment.

## Specification

### Self-Recreation Condition

The network coin puzzle always emits:

```clojure
(CREATE_COIN MY_PUZZLE_HASH MY_AMOUNT)
```

Where:
- `MY_PUZZLE_HASH`: The puzzle hash of the current network coin (unchanged)
- `MY_AMOUNT`: The amount of the current coin (1 mojo for singletons)

### Singleton Semantics

Per Chia singleton rules:
- The recreated coin has the same puzzle hash (same curried parameters)
- The amount must be odd (convention: 1 mojo)
- Exactly one successor coin is created with the singleton puzzle hash
- The singleton wrapper validates this invariant

### State Persistence

The network coin carries no mutable state — all parameters are curried at deployment:
- `REGISTRATION_COIN_MOD_HASH` (constant)
- `COLLATERAL_AMOUNT` (constant)
- `CHECKPOINT_SINGLETON_ID` (constant)

Each recreation produces an identical puzzle with a new coin ID (due to new parent).

### Continuous Availability

Because the network coin always recreates itself:
1. Registration never "uses up" the network coin
2. Multiple validators can register sequentially
3. The registration authority persists indefinitely
4. No administrative action is needed to "refill" the network coin

### Transaction Sequence

A complete registration spend bundle:

```
Spend 1: Network Coin
    - Inputs: solution with new_validator_pubkey
    - Outputs:
        - CREATE_COIN: registration coin (COLLATERAL_AMOUNT)
        - CREATE_COIN: network coin recreation (1 mojo)
        - AGG_SIG_ME: validator signature

Spend 2: Validator's XCH (funding collateral)
    - Inputs: XCH coins totaling >= COLLATERAL_AMOUNT
    - Outputs:
        - Absorbed into registration coin
        - Change back to validator
```

## Acceptance Criteria

- [ ] After registration spend, new network coin exists
- [ ] New network coin has identical puzzle hash
- [ ] New network coin has 1 mojo amount
- [ ] New network coin parent is the spent network coin
- [ ] Singleton lineage continues unbroken
- [ ] Multiple sequential registrations all succeed
- [ ] No manual intervention needed between registrations

## Implementation Notes

- **Primary codebase:** `puzzles/network_coin_inner.clsp`
- **Driver code:** `src/network_coin.rs`

### Puzzle Source (Chialisp)

```clojure
; Always recreate self
(list CREATE_COIN MY_PUZZLE_HASH MY_AMOUNT)
```

### Singleton Wrapper Handling

The `singleton_top_layer_v1_1` wrapper:
1. Verifies the inner puzzle creates exactly one coin with singleton puzzle hash
2. Wraps the recreation to maintain proper lineage
3. Ensures odd amount (coin ID tracking)

## Verification

1. Deploy network coin
2. Register validator A → network coin recreated
3. Register validator B → network coin recreated
4. Register validator C → network coin recreated
5. Verify continuous lineage: launcher → coin1 → coin2 → coin3 → coin4
6. Verify all network coins have same puzzle hash
7. Verify all network coins have 1 mojo amount
