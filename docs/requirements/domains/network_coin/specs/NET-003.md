# NET-003 — Registration Coin Creation

> **Authoritative requirement:** [NET-003](../NORMATIVE.md#NET-003)
> **Verification:** [VERIFICATION.md](../VERIFICATION.md)
> **Tracking:** [TRACKING.yaml](../TRACKING.yaml)

## Summary

When a validator registers, the network coin creates their registration coin with a specific puzzle hash and collateral amount. The puzzle hash is deterministically derived from the validator's pubkey and system parameters, ensuring the indexer can identify and verify registration coins.

## Specification

### Puzzle Hash Derivation

The registration coin puzzle hash is computed as:

```
registration_coin_puzzle_hash = curry_hash(
    REGISTRATION_COIN_MOD_HASH,
    new_validator_pubkey,
    CHECKPOINT_SINGLETON_ID
)
```

Where:
- `REGISTRATION_COIN_MOD_HASH`: Tree hash of the base registration coin puzzle (32 bytes)
- `new_validator_pubkey`: The registering validator's BLS pubkey (48 bytes)
- `CHECKPOINT_SINGLETON_ID`: Launcher ID of the checkpoint singleton (32 bytes)

### curry_hash Algorithm

Per Chialisp currying semantics:

```python
def curry_hash(mod_hash, *args):
    """Compute puzzle hash of curried puzzle"""
    result = mod_hash
    for arg in reversed(args):
        # Each curry wraps: (c (q . arg) (c (q . mod) (c (a) @)))
        arg_tree_hash = tree_hash(quote(arg))
        result = sha256(
            0x02,  # cons type
            sha256(0x01, arg_tree_hash),  # (q . arg)
            sha256(0x02, sha256(0x01, result), CURRY_TAIL_HASH)
        )
    return result
```

### CreateCoin Condition

The network coin puzzle emits:

```clojure
(CREATE_COIN registration_coin_puzzle_hash COLLATERAL_AMOUNT (pubkey))
```

Where:
- `registration_coin_puzzle_hash`: Computed as above
- `COLLATERAL_AMOUNT`: Fixed collateral amount in mojos (curried into network coin)
- `(pubkey)`: Memo list containing the validator pubkey for indexer detection

### Resulting Registration Coin

The created coin has:
- **Puzzle hash**: `registration_coin_puzzle_hash` (deterministic from pubkey)
- **Amount**: `COLLATERAL_AMOUNT` (e.g., 1 XCH = 1,000,000,000,000 mojos)
- **Parent coin ID**: The network coin's coin ID (this is the lineage proof)

### Coin ID Computation

The registration coin's coin ID is:

```
registration_coin_id = sha256(
    network_coin_coin_id,      # Parent
    registration_coin_puzzle_hash,
    COLLATERAL_AMOUNT          # As 8-byte big-endian
)
```

This coin ID is deterministic given the network coin state and validator pubkey.

## Acceptance Criteria

- [ ] Registration coin created with exactly `COLLATERAL_AMOUNT` mojos
- [ ] Puzzle hash computed using `curry_hash(MOD_HASH, pubkey, CHECKPOINT_ID)`
- [ ] Parent coin ID is the spent network coin
- [ ] Coin ID is deterministically derivable from inputs
- [ ] Indexer can verify lineage: parent is network coin spend
- [ ] Registration fails if spend bundle doesn't include required collateral

## Implementation Notes

- **Primary codebase:** `puzzles/network_coin_inner.clsp`
- **Driver code:** `src/network_coin.rs`
- **Dependencies:** `curry_hash` utility function, `REGISTRATION_COIN_MOD_HASH` constant

### Collateral Handling

The collateral comes from the validator's wallet in the same spend bundle:
1. Validator spends their XCH coin(s) totaling at least `COLLATERAL_AMOUNT`
2. Network coin creates registration coin with that amount
3. Any excess returns to validator's change address

### CLVM Cost

The CREATE_COIN condition costs approximately 1,800,000 CLVM units. The curry_hash computation is done off-chain by the driver.

## Verification

1. Compute expected puzzle hash: `curry_hash(MOD_HASH, pubkey, CHECKPOINT_ID)`
2. Spend network coin for registration
3. Query blockchain for new coin with computed puzzle hash
4. Verify coin amount equals `COLLATERAL_AMOUNT`
5. Verify parent coin ID is the spent network coin
6. Attempt registration without sufficient collateral; verify failure
