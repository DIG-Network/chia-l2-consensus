# NET-008 — Failure Case Coverage

> **Authoritative requirement:** [NET-008](../NORMATIVE.md#NET-008)
> **Verification:** [VERIFICATION.md](../VERIFICATION.md)
> **Tracking:** [TRACKING.yaml](../TRACKING.yaml)

## Summary

NET-006 has 4 simulator tests but all are success-path only. Per SCHEMA.md
Permutation Testing requirements, tests MUST cover failure cases where invalid
inputs cause spend rejection. This requirement formalises the missing negative
test coverage.

## Specification

Add to `tests/vv_req_net_006.rs` (or a new `tests/vv_req_net_008.rs`):

### Test 1: Wrong BLS signature rejected
```
1. Deploy network coin via simulator
2. Build registration spend with WRONG secret key
   (sign with sk_A but register pubkey pk_B)
3. Submit spend bundle to simulator
4. Assert: spend rejected (AGG_SIG_ME mismatch)
```

### Test 2: Insufficient collateral rejected
```
1. Deploy network coin via simulator
2. Build registration spend with collateral < COLLATERAL_AMOUNT
3. Submit spend bundle to simulator
4. Assert: spend rejected (insufficient amount)
```

### Test 3: Pubkey mismatch in AGG_SIG_ME
```
1. Deploy network coin via simulator
2. Build spend with correct signature but tampered solution pubkey
3. Submit spend bundle to simulator
4. Assert: spend rejected (signature covers different pubkey)
```

### Test 4: Verify recreated singleton puzzle hash
```
1. Deploy and register validator
2. After spend, query recreated network coin
3. Assert: recreated coin puzzle hash matches expected singleton hash
4. Assert: recreated coin amount == 1 mojo
```

## Acceptance Criteria

- [ ] Wrong BLS signature causes spend bundle rejection
- [ ] Insufficient collateral causes rejection
- [ ] Pubkey mismatch between signature and solution causes rejection
- [ ] Recreated singleton has correct puzzle hash and amount
- [ ] Each failure test verifies the spend is actually rejected (not just that it "should" fail)
