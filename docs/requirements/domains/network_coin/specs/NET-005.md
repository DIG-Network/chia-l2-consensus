# NET-005 — Pubkey Memo Convention

> **Authoritative requirement:** [NET-005](../NORMATIVE.md#NET-005)
> **Verification:** [VERIFICATION.md](../VERIFICATION.md)
> **Tracking:** [TRACKING.yaml](../TRACKING.yaml)
> **CHIP reference:** [chip-groth16-l2-consensus.md](../../../../resources/chip-groth16-l2-consensus.md) — Part 1: Network Coin

## Summary

The network coin driver MUST include the validator's pubkey as the first memo when creating the registration coin. This memo enables the indexer to detect registration coins and extract the validator pubkey without needing to decurry the puzzle on-chain.

## Specification

### Memo Requirement

The `CREATE_COIN` condition for the registration coin MUST include memos:

```clojure
(CREATE_COIN registration_coin_puzzle_hash COLLATERAL_AMOUNT (pubkey))
```

Where:
- `(pubkey)` is a list containing the 48-byte BLS pubkey as the first element
- Additional memos MAY follow but the first memo MUST be the pubkey

### Memo Format

| Position | Content | Size | Description |
|----------|---------|------|-------------|
| 0 | `pubkey` | 48 bytes | BLS12-381 G1 point, compressed, ZCash format |
| 1+ | (optional) | varies | Additional metadata (not standardized) |

### Why Memos Are Necessary

The registration coin puzzle hash is a curried hash that embeds the pubkey, but:
1. Extracting the pubkey from a curried puzzle hash requires reverse-engineering the curry
2. On-chain decurrying is expensive and complex
3. Memos are stored in the coin record and are trivially accessible via RPC

The memo convention provides an efficient channel for the indexer to discover:
- Which coins are registration coins
- Which pubkey each registration coin belongs to

### Indexer Usage

The indexer processes `CREATE_COIN` conditions from network coin spends:

```python
def process_network_coin_spend(spend):
    for condition in spend.conditions:
        if condition.op == CREATE_COIN:
            puzzle_hash, amount, memos = condition.args

            if amount == COLLATERAL_AMOUNT and len(memos) > 0:
                pubkey = memos[0]
                if len(pubkey) == 48:
                    # Valid registration coin
                    verify_puzzle_hash(puzzle_hash, pubkey)
                    register_validator(pubkey, coin_id)
```

### Driver Implementation

The driver code (not the puzzle) is responsible for constructing the memo list:

```rust
// In network_coin driver
fn build_registration_conditions(pubkey: &PublicKey) -> Vec<Condition> {
    let puzzle_hash = compute_registration_puzzle_hash(pubkey);

    vec![
        Condition::CreateCoin {
            puzzle_hash,
            amount: COLLATERAL_AMOUNT,
            memos: vec![pubkey.to_bytes().to_vec()],  // Pubkey as first memo
        },
        // ... other conditions
    ]
}
```

### Puzzle vs Driver Responsibility

| Concern | Puzzle | Driver |
|---------|--------|--------|
| Verify pubkey ownership | ✓ (AGG_SIG_ME) | |
| Compute puzzle hash | ✓ (curry_hash) | |
| Set collateral amount | ✓ (curried param) | |
| Include pubkey memo | | ✓ (memo list construction) |

The puzzle enforces correctness; the driver provides convenience.

## Acceptance Criteria

- [ ] `CREATE_COIN` for registration coin includes memo list
- [ ] First memo is exactly 48 bytes (BLS pubkey)
- [ ] Memo content matches the pubkey in the puzzle hash curry
- [ ] Indexer can extract pubkey from memo without decurrying
- [ ] Registration coins without pubkey memo are still valid but may not be indexed

## Implementation Notes

- **Primary codebase:** `src/network_coin.rs` (driver code)
- **Not enforced by puzzle:** This is a convention, not a consensus rule
- **Backwards compatibility:** Old coins without memos require puzzle decurrying

### Why Not Enforce in Puzzle?

Making the memo mandatory in the puzzle would:
1. Increase puzzle complexity and CLVM cost
2. Create potential for memo/puzzle hash mismatches
3. Add no security benefit (the pubkey is already committed in the puzzle hash)

The memo is a convenience for indexing, not a security mechanism.

## Verification

1. Register validator with network coin driver
2. Query the `CREATE_COIN` condition from the spend
3. Verify memos list is non-empty
4. Verify first memo is 48 bytes
5. Verify memo content matches the pubkey used in registration
6. Verify indexer correctly extracts pubkey from memo
7. Register without memo (raw puzzle spend); verify coin exists but indexer warns
