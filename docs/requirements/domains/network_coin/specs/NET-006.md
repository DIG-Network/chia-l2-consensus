# NET-006 — End-to-End Simulator Test

> **Authoritative requirement:** [NET-006](../NORMATIVE.md#NET-006)
> **Verification:** [VERIFICATION.md](../VERIFICATION.md)
> **Tracking:** [TRACKING.yaml](../TRACKING.yaml)

## Summary

All prior NET requirements (NET-001 through NET-005) were verified using CLVM execution tests and source inspection. This requirement mandates a full end-to-end simulator test that exercises the network coin as a complete singleton — wrapping the inner puzzle via chia-wallet-sdk, submitting real spend bundles to the simulator, and verifying resulting coin states match expectations.

The chia-wallet-sdk handles singleton wrapping (the inner puzzle is wrapped with `singleton_top_layer_v1_1` at runtime, not in Rue). This test proves the inner puzzle integrates correctly with the SDK's singleton layer.

## Specification

### Test Lifecycle

```
1. DEPLOY NETWORK COIN
   - Create genesis coin in simulator
   - Wrap network_coin_inner puzzle with chia-wallet-sdk Singleton
   - Launch singleton via simulator
   - Verify: exactly one network coin exists with 1 mojo (odd amount)
   - Verify: launcher coin is spent

2. REGISTER VALIDATOR
   - Generate BLS keypair (sk, pk) via test_secret_key()
   - Compute registration message: sha256("register" + pk)
   - Build spend bundle:
     * Network coin spend with singleton wrapper
     * Solution: (new_validator_pubkey, conditions)
     * Funding coin for collateral
   - Sign with validator's secret key (AGG_SIG_ME)
   - Submit spend bundle to simulator
   - Verify: network coin recreated (new coin ID, same puzzle hash structure)
   - Verify: registration coin created with:
     * Puzzle hash = curry_hash(REG_MOD_HASH, pk, CHECKPOINT_ID)
     * Amount = COLLATERAL_AMOUNT
   - Verify: pubkey memo present on registration coin creation

3. SEQUENTIAL REGISTRATIONS
   - Register a second validator using the recreated network coin
   - Verify: both registration coins exist
   - Verify: network coin recreated again with continuous lineage

4. FAILURE CASES
   - Attempt registration with wrong secret key → spend rejected
   - Attempt registration without sufficient collateral → spend rejected
   - Attempt to spend network coin as non-singleton → spend rejected
```

### Singleton Wrapping Pattern

The network coin inner puzzle must be wrapped at runtime:

```rust
use chia_sdk_driver::{Singleton, SpendContext};
use chia_sdk_test::Simulator;

// Load inner puzzle
let inner_puzzle = node_from_bytes(&mut allocator, &NETWORK_COIN_INNER_HEX)?;

// Curry inner puzzle with parameters
let curried = CurriedProgram {
    program: inner_puzzle,
    args: &NetworkCoinArgs {
        registration_coin_mod_hash: REG_MOD_HASH,
        collateral_amount: COLLATERAL_AMOUNT,
        checkpoint_singleton_id: CHECKPOINT_ID,
    },
};

// chia-wallet-sdk wraps with singleton_top_layer_v1_1
// and handles lineage, odd-amount, and coin ID tracking
```

### Specific Assertions

After a successful registration spend:

```rust
// Network coin recreated
let new_network_coins = sim.coin_records_by_puzzle_hash(network_coin_puzzle_hash);
assert_eq!(new_network_coins.len(), 1);
assert_eq!(new_network_coins[0].coin.amount, 1);  // Singleton odd amount

// Registration coin created
let reg_coins = sim.coin_records_by_puzzle_hash(reg_puzzle_hash);
assert_eq!(reg_coins.len(), 1);
assert_eq!(reg_coins[0].coin.amount, COLLATERAL_AMOUNT);

// Old network coin spent
assert!(sim.coin_state(old_network_coin_id).is_spent());

// Lineage: new network coin's parent = old network coin
assert_eq!(new_network_coins[0].coin.parent_coin_info, old_network_coin_id);
```
