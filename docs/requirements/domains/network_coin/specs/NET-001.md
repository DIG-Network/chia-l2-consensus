# NET-001 — Network Coin Singleton Identity

> **Authoritative requirement:** [NET-001](../NORMATIVE.md#NET-001)
> **Verification:** [VERIFICATION.md](../VERIFICATION.md)
> **Tracking:** [TRACKING.yaml](../TRACKING.yaml)
> **CHIP reference:** [chip-groth16-l2-consensus.md](../../../../resources/chip-groth16-l2-consensus.md) — Part 1: Network Coin

## Summary

The network coin is the canonical registration authority for the entire L2 network. It MUST be implemented as a Chia singleton to guarantee exactly one instance exists. Without this singleton property, attackers could create arbitrary registration coins by deploying fake "network coins" and spending them.

The singleton wrapper enforces that:
1. Only one network coin can exist with a given launcher ID
2. Every spend recreates exactly one successor coin
3. The lineage from launcher to current coin is verifiable on-chain

## Specification

### Singleton Structure

The network coin uses the standard Chia singleton pattern:

```
outer_puzzle = singleton_top_layer_v1_1(
    SINGLETON_STRUCT,        # (MOD_HASH, LAUNCHER_ID, LAUNCHER_PUZZLE_HASH)
    inner_puzzle             # network_coin_inner.clsp
)
```

**Singleton parameters:**
- `LAUNCHER_ID`: Coin ID of the launcher coin (32 bytes)
- `MOD_HASH`: Tree hash of singleton_top_layer_v1_1 module
- `LAUNCHER_PUZZLE_HASH`: Standard singleton launcher puzzle hash

### Inner Puzzle Curried Parameters

| Parameter | Type | Description |
|-----------|------|-------------|
| `REGISTRATION_COIN_MOD_HASH` | Bytes32 | Tree hash of registration coin puzzle (before currying) |
| `COLLATERAL_AMOUNT` | u64 | Required collateral in mojos |
| `CHECKPOINT_SINGLETON_ID` | Bytes32 | Launcher ID of checkpoint singleton |

### Deployment

The network coin is created once per L2 network during initial deployment:

1. Spend a genesis coin to create the singleton launcher
2. Spend the launcher to create the first network coin
3. Record the `LAUNCHER_ID` in network configuration

The launcher ID becomes the permanent identifier for this L2 network's registration authority.

### Uniqueness Guarantee

The singleton pattern prevents duplicate network coins because:
- The launcher can only be spent once (it's a regular coin)
- Each spend creates exactly one successor with odd amount
- The singleton wrapper validates lineage back to the launcher

Any coin claiming to be a network coin without valid singleton lineage is invalid and MUST be ignored by the indexer.

## Acceptance Criteria

- [ ] Network coin puzzle uses `singleton_top_layer_v1_1` wrapper
- [ ] Launcher ID is derived from genesis coin spend
- [ ] Only one network coin can exist at any time
- [ ] Attempt to create duplicate network coin fails
- [ ] Singleton lineage is verifiable from any network coin spend

## Implementation Notes

- **Puzzle source:** `puzzles/network_coin_inner.rue` (Rue)
- **Compiled hex:** `puzzles/compiled/network_coin_inner.hex` (CLVM bytecode)
- **Compiled hash:** `puzzles/compiled/network_coin_inner.hash` (tree hash)
- **Driver code:** `src/puzzles/network_coin.rs`
- **Dependencies:** Chia singleton standard (`singleton_top_layer_v1_1`)

### Compile Pipeline

The Rue source is compiled to `.hex` (CLVM bytecode) and `.hash` (tree hash),
which are embedded into the Rust driver via `include_str!`:

```
puzzles/network_coin_inner.rue
   │  rue build -x  → puzzles/compiled/network_coin_inner.hex
   │  rue build --hash → puzzles/compiled/network_coin_inner.hash
   ▼
src/puzzles/network_coin.rs
   NETWORK_COIN_INNER_PUZZLE_HEX  = include_str!("...hex")
   NETWORK_COIN_INNER_MOD_HASH_HEX = include_str!("...hash")
```

## Verification

1. Deploy network coin via genesis spend
2. Query blockchain for coins with network coin puzzle hash
3. Verify exactly one coin exists
4. Attempt to create a second network coin with same parameters
5. Verify the second creation fails or is rejected
6. Verify singleton lineage from current coin back to launcher
