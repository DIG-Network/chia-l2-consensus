# NET-002 — AggSigMe Registration Verification

> **Authoritative requirement:** [NET-002](../NORMATIVE.md#NET-002)
> **Verification:** [VERIFICATION.md](../VERIFICATION.md)
> **Tracking:** [TRACKING.yaml](../TRACKING.yaml)

## Summary

When a validator registers, the network coin MUST verify they control the claimed BLS pubkey. This prevents one party from registering on behalf of another without authorization. The verification uses Chia's `AGG_SIG_ME` condition with a deterministic message format.

## Specification

### Signature Message Format

The registration message is constructed as:

```
registration_message = sha256("register" + pubkey)
```

Where:
- `"register"` is the literal UTF-8 string (8 bytes, no null terminator)
- `pubkey` is the 48-byte BLS12-381 G1 point (compressed, ZCash format)
- Total input to sha256: 56 bytes
- Output: 32-byte hash

### AGG_SIG_ME Condition

The network coin puzzle emits:

```clojure
(AGG_SIG_ME new_validator_pubkey registration_message)
```

The full signing message becomes (per Chia's AGG_SIG_ME semantics):
```
signing_message = registration_message + genesis_challenge + network_coin_coin_id
```

Where:
- `genesis_challenge`: 32-byte Chia network identifier
- `network_coin_coin_id`: 32-byte ID of the current network coin being spent

### Signature Verification

The spend bundle's aggregate signature must include a valid BLS signature:
```
sig = BLS_Sign(validator_secret_key, signing_message)
```

If the signature is invalid or missing, the spend is rejected by the Chia node.

### Security Properties

This requirement ensures:

1. **Key ownership**: Only the holder of the private key can register the corresponding pubkey
2. **Replay protection**: The message includes the coin ID, so a valid signature cannot be replayed across different network coin spends
3. **Network binding**: The genesis challenge binds the signature to a specific Chia network (mainnet vs testnet)

### Pseudocode

```rust
// In network_coin_inner puzzle
fn main(
    REGISTRATION_COIN_MOD_HASH: Bytes32,
    COLLATERAL_AMOUNT: Int,
    CHECKPOINT_SINGLETON_ID: Bytes32,
    new_validator_pubkey: PublicKey,
    conditions: List<Condition>,
) -> List<Condition> {

    let registration_message = sha256(b"register" + new_validator_pubkey);

    conditions + [
        AggSigMe(new_validator_pubkey, registration_message),
        // ... other conditions
    ]
}
```

## Acceptance Criteria

- [ ] Registration message format: `sha256("register" + pubkey)` exactly
- [ ] AGG_SIG_ME condition emitted with correct pubkey and message
- [ ] Spend with valid signature from matching key succeeds
- [ ] Spend with signature from different key is rejected
- [ ] Spend with no signature is rejected
- [ ] Spend with malformed pubkey is rejected
- [ ] Message does not change based on solution fields (deterministic from pubkey only)

## Implementation Notes

- **Primary codebase:** `puzzles/network_coin_inner.clsp`
- **Driver code:** `src/network_coin.rs` — `register_validator()` function
- **Dependencies:** BLS12-381 signing library

### CLVM Cost

The AGG_SIG_ME condition costs approximately 1,200,000 CLVM units. Combined with sha256 and other operations, the total registration spend is ~5,300,000 units.

## Verification

1. Generate BLS keypair (sk, pk)
2. Compute registration_message = sha256("register" + pk)
3. Sign: sig = BLS_Sign(sk, registration_message + genesis_challenge + coin_id)
4. Build spend bundle with network coin spend + signature
5. Submit to node; verify acceptance
6. Repeat with wrong key; verify rejection
7. Repeat with no signature; verify rejection
