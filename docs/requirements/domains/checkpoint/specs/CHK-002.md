# CHK-002 — State Tracking

> **Authoritative requirement:** [CHK-002](../NORMATIVE.md#CHK-002)
> **Verification:** [VERIFICATION.md](../VERIFICATION.md)
> **Tracking:** [TRACKING.yaml](../TRACKING.yaml)

## Summary

The checkpoint singleton tracks four state values that together represent the complete L2 consensus state. These values are curried into the inner puzzle and change with each checkpoint spend. The state is the authoritative record of L2 consensus.

## Specification

### State Values

| Field | Type | Size | Description |
|-------|------|------|-------------|
| `state_root` | Bytes32 | 32 bytes | L2 state rollup hash (application-defined) |
| `epoch` | u64 | 8 bytes | Monotonic counter, increments by 1 per checkpoint |
| `validator_merkle_root` | Bytes32 | 32 bytes | Sparse Merkle tree root of active validators |
| `validator_count` | u64 | 8 bytes | Number of active validators |

### state_root

- Application-defined hash of L2 state
- Could be: transaction root, account root, UTXO set hash, etc.
- Opaque to the checkpoint singleton (not validated)
- Committed to by majority validator signature

### epoch

- Starts at 0 on deployment
- Increments by exactly 1 on each checkpoint spend
- Never decrements (monotonic)
- Used for:
  - Replay protection in membership announcements
  - Checkpoint ordering/sequencing
  - State history tracking

### validator_merkle_root

- Root of sparse Merkle tree ([SMT-001](../../smt/specs/SMT-001.md))
- Changes when validators join or leave
- Verified by majority signature over checkpoint message
- Used for membership/non-membership proofs

### validator_count

- Total number of active validators
- Used in majority calculation: 2k > validator_count
- Changes when validators join or leave
- Runtime input to Groth16 circuit (not compiled in)

### Curry Structure

```clojure
(mod (
    ; Curried in (state)
    STATE_ROOT
    EPOCH
    VALIDATOR_MERKLE_ROOT
    VALIDATOR_COUNT

    ; Curried in (constants)
    VK
    TREE_DEPTH
    EMPTY_LEAF_HASH

    ; Solution
    spend_path
    ...
)
    ; Puzzle logic
)
```

### State Transitions

**Checkpoint spend:**
```
Before: (state_root_old, epoch_old, merkle_root_old, count_old)
After:  (state_root_new, epoch_old+1, merkle_root_new, count_new)
```

**Membership query spend:**
```
Before: (state_root, epoch, merkle_root, count)
After:  (state_root, epoch, merkle_root, count)  # UNCHANGED
```

### Initial State

At deployment (epoch 0):
```
state_root = 0x00...00 (or application-defined genesis)
epoch = 0
validator_merkle_root = EMPTY_TREE_ROOT
validator_count = 0
```

Where `EMPTY_TREE_ROOT` is the Merkle root of a tree with all empty leaves.

### Puzzle Hash Changes

Because state is curried, the puzzle hash changes with each checkpoint:
```
puzzle_hash = curry_hash(
    CHECKPOINT_INNER_MOD_HASH,
    state_root, epoch, merkle_root, count,
    VK, TREE_DEPTH, EMPTY_LEAF_HASH
)
```

The outer singleton wrapper puzzle hash remains constant (based on launcher ID).

## Acceptance Criteria

- [ ] Four state values curried into inner puzzle
- [ ] `state_root` is 32 bytes
- [ ] `epoch` is u64, starts at 0, increments by 1
- [ ] `validator_merkle_root` is 32 bytes
- [ ] `validator_count` is u64
- [ ] Checkpoint spend updates all four values
- [ ] Membership query spend preserves all values unchanged
- [ ] State values accessible via decurrying

## Implementation Notes

- **Primary codebase:** `puzzles/checkpoint_inner.clsp`
- **State accessors:** `src/checkpoint.rs` — `state()`, `epoch()`, etc.

### Querying Current State

```rust
let checkpoint = client.sync().checkpoint_state();
println!("Epoch: {}", checkpoint.epoch);
println!("State root: {}", hex::encode(checkpoint.state_root));
println!("Validator count: {}", checkpoint.validator_count);
```

### State Consistency

The indexer verifies state consistency by:
1. Computing Merkle root from known registration coins
2. Comparing to `validator_merkle_root` from checkpoint
3. Failing with `StateMismatch` if different

## Verification

1. Deploy checkpoint singleton with initial state
2. Decurry; verify 4 state values present
3. Verify initial: epoch=0, count=0, merkle_root=empty
4. Submit checkpoint with new values
5. Decurry new singleton; verify state updated
6. Verify epoch = previous_epoch + 1
7. Submit membership query; verify state unchanged
