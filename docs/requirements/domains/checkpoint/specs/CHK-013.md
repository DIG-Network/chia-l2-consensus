# CHK-013 — Validator Attestation Binding

> **Authoritative requirement:** [CHK-013](../NORMATIVE.md#CHK-013)
> **Verification:** [VERIFICATION.md](../VERIFICATION.md)
> **Tracking:** [TRACKING.yaml](../TRACKING.yaml)
> **CHIP reference:** [chip-groth16-l2-consensus.md](../../../../resources/chip-groth16-l2-consensus.md) — Security

## Summary

Validators must sign a message that explicitly commits to the epoch number,
network identity, and L2 state hash. The combined Groth16 + bls_verify
checks prove that a majority of legitimate validators attested to this
specific (epoch, network, state) tuple.

## Specification

### The Attestation Message

Validators sign a message derived from the checkpoint_message, which is:

```
checkpoint_message = sha256(
    new_state_root              (32 bytes)  ← state hash attestation
    + new_validator_merkle_root (32 bytes)
    + new_validator_count       (8 bytes)
    + new_epoch                 (8 bytes)   ← epoch attestation
    + network_coin_launcher_id  (32 bytes)  ← network identity attestation
)
```

The full validator signing message is:
```
signing_message = checkpoint_message + genesis_challenge + checkpoint_coin_id
```

### The Three-Part Proof

1. **BLS Signature Verification** (CLVM `bls_verify`, opcode 59):
   Proves the aggregate public key `agg_signers` actually signed the
   checkpoint_message containing epoch + network_id + state_hash.

2. **Groth16 Proof Verification** (CLVM `bls_pairing_identity`, opcode 58):
   Proves `agg_signers` is the G1 sum of k registered validators whose
   pubkeys are in the Merkle tree, and 2k > validator_count.

3. **Checkpoint Message Binding** (CLVM scalar assertion):
   The puzzle recomputes checkpoint_message from its curried state
   (epoch, network_id) and solution (state_root), verifies the scalar
   matches, ensuring the proof is bound to the actual message.

### What This Proves End-to-End

Together, these three checks prove:
- A majority of registered validators (k > n/2)
- Whose pubkeys are in the Merkle tree (membership verified)
- Actually signed (`bls_verify` passes)
- A message containing the specific epoch, network ID, and state hash
- That was recomputed by the puzzle from on-chain state (not supplied by attacker)

### Why Each Field Matters

| Field | Prevents | Binding Mechanism |
|-------|----------|-------------------|
| `new_epoch` | Replay across epochs | Puzzle computes `old_epoch + 1` internally (CHK-009) |
| `network_coin_launcher_id` | Cross-network replay | Curried into puzzle at deployment (CHK-012) |
| `new_state_root` | Arbitrary state submission | Included in checkpoint_message hash (CHK-011) |

## Acceptance Criteria

- [ ] checkpoint_message includes all three fields (epoch, network_id, state_hash)
- [ ] Validators sign a message derived from the checkpoint_message
- [ ] CLVM bls_verify verifies signature over the checkpoint_message
- [ ] Groth16 proof proves signers form a legitimate majority
- [ ] Changing any of the three fields invalidates the signature
- [ ] Puzzle recomputes the checkpoint_message (not from solution)
- [ ] Test: different epoch → different signature → bls_verify fails
- [ ] Test: different network → different message → signature invalid
- [ ] Test: different state_root → different message → signature invalid
