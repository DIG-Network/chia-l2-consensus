# CHK-009 — Epoch Binding

> **Authoritative requirement:** [CHK-009](../NORMATIVE.md#CHK-009)
> **Verification:** [VERIFICATION.md](../VERIFICATION.md)
> **Tracking:** [TRACKING.yaml](../TRACKING.yaml)
> **CHIP reference:** [chip-groth16-l2-consensus.md](../../../../resources/chip-groth16-l2-consensus.md) — Part 3: Checkpoint Singleton

## Summary

The epoch is bound to the Groth16 proof through the checkpoint_message hash.
The CLVM puzzle computes `new_epoch = old_epoch + 1` internally and
recomputes the checkpoint_message using that epoch. A proof generated for
epoch N cannot verify at epoch M because the checkpoint_message differs,
causing the scalar assertion to fail.

## Specification

### Binding Chain

```
CLVM puzzle state: epoch (curried)
    ↓
new_epoch = epoch + 1                          (puzzle computes, line 265)
    ↓
checkpoint_message = sha256(
    new_state_root ‖ new_merkle_root ‖
    new_validator_count_be8 ‖ new_epoch_be8     (epoch embedded in hash)
)                                               (puzzle computes, lines 273-277)
    ↓
scalar s6 = sha256(checkpoint_message)          (puzzle verifies, line 297)
    ↓
Groth16 proof verified with checkpoint_message   (public input #6)
    ↓
BLS signature verified over checkpoint_message   (puzzle verifies, line 342)
```

### Why Epoch Cannot Be Spoofed

1. **Puzzle computes new_epoch internally** — the solution cannot supply it
2. **Puzzle recomputes checkpoint_message** — includes the computed new_epoch
3. **Scalar assertion** — s6 must equal sha256(recomputed checkpoint_message)
4. **If epoch differs** — checkpoint_message differs → s6 mismatches → spend fails
5. **BLS signature** — validators signed a specific checkpoint_message containing the epoch

### Circuit Side

The circuit receives `checkpoint_message` as public input #6. The circuit
does not decompose the hash (SHA-256 is one-way), but the binding is
enforced externally:

- The puzzle supplies the checkpoint_message hash to the verifier
- The puzzle verifies the hash matches its own computation (which includes epoch)
- The circuit verifies the proof against this specific hash value

### What This Prevents

- **Proof replay**: A proof for epoch 5 cannot be used at epoch 10
- **Epoch skipping**: New_epoch must be exactly old_epoch + 1
- **Epoch regression**: Can only go forward (old_epoch + 1)

## Acceptance Criteria

- [ ] Checkpoint_message includes new_epoch in its sha256 preimage (80-byte input)
- [ ] CLVM puzzle computes new_epoch = old_epoch + 1 (not from solution)
- [ ] CLVM puzzle recomputes checkpoint_message using the computed new_epoch
- [ ] Scalar assertion binds the proof to the specific checkpoint_message
- [ ] A proof generated for epoch N fails when puzzle is at epoch M ≠ N-1
- [ ] The Rue source does NOT accept epoch from the solution
