# CHK-015 — CLVM Execution for Binding Properties

> **Authoritative requirement:** [CHK-015](../NORMATIVE.md#CHK-015)
> **Verification:** [VERIFICATION.md](../VERIFICATION.md)
> **Tracking:** [TRACKING.yaml](../TRACKING.yaml)

## Summary

CHK-009 through CHK-014 (53 tests total) verify cryptographic binding
properties through Rust-side computation only. While CHK-008 provides E2E
integration coverage, dedicated CLVM execution tests would strengthen confidence
that the compiled puzzle itself enforces these bindings.

## Specification

### Minimum CLVM execution tests

1. **Epoch binding (CHK-009):** Run checkpoint path CLVM with two different
   epochs. Verify the output announcement contains different epoch values.
   Verify new_epoch = old_epoch + 1 in recreation puzzle hash.

2. **Network ID binding (CHK-012):** Run checkpoint path CLVM with two
   different network_coin_launcher_id values curried in. Verify the checkpoint
   messages differ.

3. **Invalid proof rejection (CHK-014):** Run checkpoint path CLVM with
   corrupted proof data. Verify the puzzle execution fails/errors rather than
   producing valid conditions.

### Implementation approach

Use the existing `tests/common/clvm.rs` helpers:
- `load_puzzle()` → deserialize checkpoint_inner.hex
- `curry_puzzle()` → curry with test VK, state, etc.
- `run_puzzle()` → execute and check result

## Acceptance Criteria

- [ ] ≥1 CLVM execution test for epoch binding
- [ ] ≥1 CLVM execution test for network_id binding
- [ ] ≥1 CLVM execution test for invalid proof rejection
- [ ] Tests use compiled `.hex` artifact, not source inspection
