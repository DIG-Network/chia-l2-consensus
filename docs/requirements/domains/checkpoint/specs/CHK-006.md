# CHK-006 — Permissionless Membership Query

> **Authoritative requirement:** [CHK-006](../NORMATIVE.md#CHK-006)

> **Tracking:** [TRACKING.yaml](../TRACKING.yaml)

## Summary

The membership query spend path requires no signature or special authorization. Anyone can query membership status at any time by providing a valid Merkle proof. This ensures validators can always recover their collateral without relying on any third party.

## Specification

### No AGG_SIG Conditions

The membership query path emits NO signature conditions:

```clojure
; Membership query output (no AGG_SIG_ME, no AGG_SIG_UNSAFE)
(list
    (CREATE_COIN_ANNOUNCEMENT membership_announcement)
    (CREATE_COIN MY_PUZZLE_HASH MY_AMOUNT))
```

### Spend Bundle Signature

Because there are no signature conditions, the spend bundle can use the identity signature (G2 point at infinity):

```rust
let spend_bundle = SpendBundle {
    coin_spends: vec![membership_query_spend],
    aggregated_signature: G2Affine::identity(), // No signature needed
};
```

### Why Permissionless

1. **Self-sovereignty**: Validators shouldn't need permission to exit
2. **Censorship resistance**: No party can block collateral recovery
3. **Simplicity**: No key management for queries
4. **Efficiency**: No signature verification cost

### What IS Required

While no signature is needed, the spender must provide:
- Valid Merkle proof (membership or non-membership)
- Correct pubkey
- Correct is_member flag matching the proof

The Merkle proof is the authorization — only someone who can construct a valid proof can successfully spend.

### Security Implications

**Cannot forge membership status:**
- To claim someone IS a member: need valid inclusion proof
- To claim someone is NOT a member: need valid exclusion proof
- Both require knowledge of the Merkle tree structure

**Cannot prevent queries:**
- Anyone with the proof can query
- No coordinator approval needed
- No timing restrictions

**Cannot change state:**
- Membership query never changes state
- Epoch, root, count all preserved
- Can be called unlimited times

### Who Can Call

| Caller | Can query? | Use case |
|--------|-----------|----------|
| Validator themselves | ✓ | Collateral recovery |
| Other validators | ✓ | Verify peer status |
| L2 operators | ✓ | State verification |
| Random third party | ✓ | Audit, transparency |
| Malicious actor | ✓ | No harm possible |

### Comparison with Checkpoint Path

| Aspect | Checkpoint | Membership Query |
|--------|-----------|------------------|
| Signature required | Yes (majority) | No |
| State changes | Yes | No |
| Proof required | Groth16 | Merkle |
| Who can call | Checkpoint submitter | Anyone |
| Cost | ~17M CLVM | ~4M CLVM |

### Spam Considerations

While permissionless, spam is naturally limited:
1. Transaction fees apply (mempool fee market)
2. Block space is limited
3. No state change means no persistent impact
4. Each query costs the spammer ~4M CLVM

Practical spam protection is provided by Chia's standard mempool policies.

## Acceptance Criteria

- [ ] No AGG_SIG_ME or AGG_SIG_UNSAFE conditions emitted
- [ ] Spend succeeds with identity signature
- [ ] Any party can submit membership query
- [ ] No special keys or permissions required
- [ ] Valid Merkle proof is sole requirement
- [ ] Invalid proof still fails (not THAT permissionless)

## Implementation Notes

- **Puzzle source:** `puzzles/checkpoint_inner.clsp`
- **Driver code:** `src/checkpoint.rs` — `query_membership()`

### Building a Permissionless Spend

```rust
fn build_membership_query_bundle(
    checkpoint_coin: &Coin,
    pubkey: &PublicKey,
    is_member: bool,
    proof: &MerkleProof,
) -> SpendBundle {
    let puzzle = compute_checkpoint_puzzle(...);
    let solution = MembershipQuerySolution {
        spend_path: 1,
        query_pubkey: pubkey.clone(),
        is_member,
        merkle_siblings: proof.siblings.clone(),
    };

    SpendBundle {
        coin_spends: vec![CoinSpend {
            coin: checkpoint_coin.clone(),
            puzzle_reveal: puzzle,
            solution: solution.to_clvm(),
        }],
        aggregated_signature: G2Affine::identity(), // No signature!
    }
}
```

## Verification

1. Build membership query spend bundle
2. Set aggregated_signature to identity
3. Submit to mempool → verify acceptance
4. Verify spend succeeds on-chain
5. Verify no signature was verified
6. Have third party (no keys) submit query → verify success
7. Build invalid proof query → verify failure (proof matters)
