# CHK-011 — State Hash Binding

> **Authoritative requirement:** [CHK-011](../NORMATIVE.md#CHK-011)
> **Verification:** [VERIFICATION.md](../VERIFICATION.md)
> **Tracking:** [TRACKING.yaml](../TRACKING.yaml)
> **CHIP reference:** [chip-groth16-l2-consensus.md](../../../../resources/chip-groth16-l2-consensus.md) — Part 3: Checkpoint Singleton

## Summary

The state_root is the L2 state hash that validators attest to. It MUST be
embedded in the checkpoint_message so that the Groth16 proof and BLS
signature are cryptographically bound to a specific state transition.
Without this binding, an attacker could submit an arbitrary state_root
with a valid proof/signature generated for a different state.

## Specification

### Current Binding Chain

```
state_root (from solution)
    ↓
checkpoint_message = sha256(
    state_root ‖ merkle_root ‖ count ‖ epoch    ← state_root is first
)
    ↓
scalar s6 = sha256(checkpoint_message)           (puzzle verifies)
    ↓
Groth16 proof public input #6 = checkpoint_message
    ↓
BLS signature over checkpoint_message
```

### What This Ensures

- Validators signed a message committing to this specific state_root
- The Groth16 proof was generated with this checkpoint_message
- Changing state_root changes checkpoint_message, breaking both checks
- The puzzle uses the same state_root for recreation (new singleton state)

### What to Verify

1. `compute_checkpoint_message()` takes state_root as first parameter
2. Rue puzzle includes state_root in sha256 preimage for checkpoint_message
3. Puzzle uses the SAME state_root in both message computation and CREATE_COIN
4. Different state_roots produce different checkpoint_messages

## Acceptance Criteria

- [ ] state_root is the first field in checkpoint_message sha256 preimage
- [ ] Different state_roots produce different checkpoint_messages
- [ ] Puzzle uses same state_root in message AND singleton recreation
- [ ] Changing state_root invalidates both proof and signature
- [ ] Rue source confirms state_root in checkpoint_message computation
