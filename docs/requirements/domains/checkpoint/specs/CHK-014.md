# CHK-014 — Permissionless Submission / Forgery Resistance

> **Authoritative requirement:** [CHK-014](../NORMATIVE.md#CHK-014)
> **Verification:** [VERIFICATION.md](../VERIFICATION.md)
> **Tracking:** [TRACKING.yaml](../TRACKING.yaml)
> **CHIP reference:** [chip-groth16-l2-consensus.md](../../../../resources/chip-groth16-l2-consensus.md) — Security

## Summary

The checkpoint system is designed so that:

1. **Anyone can submit** — no special authorization needed to spend the
   checkpoint singleton via the checkpoint path. There is no AGG_SIG_ME
   on the submitter. This enables decentralized checkpoint submission.

2. **But nobody can forge** — the puzzle rejects any submission where the
   Groth16 proof or BLS aggregate signature is invalid. A submitter who
   does not have valid off-chain signatures from a majority of registered
   validators cannot produce an accepted checkpoint.

## Threat Model

### Attacker Capabilities
- Can observe the blockchain and mempool
- Can construct arbitrary CLVM solutions
- Can submit spend bundles to any Chia node
- Does NOT control a majority of validator private keys
- Does NOT have the Groth16 proving key toxic waste

### What the Attacker CANNOT Do
1. **Forge a Groth16 proof** — requires toxic waste from trusted setup
2. **Forge BLS signatures** — requires majority of validator secret keys
3. **Supply a fake checkpoint_message** — puzzle recomputes it internally
4. **Choose an arbitrary epoch** — puzzle computes `old_epoch + 1`
5. **Forge a different network** — `network_coin_launcher_id` is curried

### Why Permissionless Submission Is Safe

The checkpoint path emits NO signature conditions (no AGG_SIG_ME,
no AGG_SIG_UNSAFE). Anyone can submit the spend bundle. Security comes
entirely from:

1. **bls_pairing_identity** (opcode 58): Verifies the Groth16 proof,
   which proves the aggregate pubkey is the G1 sum of k registered
   validators forming a strict majority (2k > count).

2. **bls_verify** (opcode 59): Verifies the BLS aggregate signature
   over the checkpoint_message, which the puzzle recomputed from
   curried state + solution parameters.

If either check fails, the CLVM puzzle raises an exception and the
spend is rejected.

### The L2 Off-Chain Flow

```
L2 Network:
  1. Validators observe L2 state transitions
  2. Each validator signs: sha256(state ‖ merkle ‖ count ‖ epoch ‖ network_id)
  3. Signatures are collected by any party (coordinator, validator, observer)
  4. Collector aggregates k signatures (where 2k > validator_count)
  5. Collector generates Groth16 proof (anyone with proving key can do this)
  6. Collector submits checkpoint spend to Chia L1

L1 Checkpoint Puzzle:
  1. Recomputes checkpoint_message from curried state + solution
  2. Verifies Groth16 proof (proves majority membership)
  3. Verifies BLS aggregate signature (proves majority signed)
  4. If both pass → epoch advances, state updates
  5. If either fails → spend rejected, nothing changes
```

## Acceptance Criteria

- [ ] No AGG_SIG_ME or AGG_SIG_UNSAFE in the checkpoint path
- [ ] Checkpoint path outputs only [recreate, announce] (no sig conditions)
- [ ] Invalid Groth16 proof → spend rejected
- [ ] Invalid BLS signature → spend rejected
- [ ] Minority signers → proof generation fails (cannot produce valid proof)
- [ ] Puzzle computes checkpoint_message internally (not from solution)
- [ ] Epoch computed internally (not from solution)
- [ ] Network ID curried (not from solution)
- [ ] Valid proof + valid signature from majority → spend accepted
