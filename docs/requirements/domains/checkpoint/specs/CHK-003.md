# CHK-003 — Groth16 and BLS Verification

> **Authoritative requirement:** [CHK-003](../NORMATIVE.md#CHK-003)
> **Verification:** [VERIFICATION.md](../VERIFICATION.md)
> **Tracking:** [TRACKING.yaml](../TRACKING.yaml)

## Summary

The checkpoint spend path performs two cryptographic verifications that together prove majority validator consensus: a Groth16 zero-knowledge proof that k pubkeys are registered validators and form a majority, and a BLS signature verification that those validators actually signed the checkpoint message.

## Specification

### Two-Check Design

| Check | What it proves | CLVM operator |
|-------|---------------|---------------|
| Groth16 proof | k pubkeys ∈ Merkle tree, 2k > count, sum = agg_signers | `bls_pairing_identity` |
| BLS signature | agg_signers signed checkpoint_message | `bls_verify` |

Neither check alone is sufficient. Together they prove:
- A legitimate majority of registered validators
- Actually signed this specific checkpoint

### Groth16 Verification

The Groth16 proof `(A, B, C)` is verified by the pairing equation:

```
e(A, B) · e(-vk_alpha, vk_beta) · e(-vk_input, vk_gamma) · e(-C, vk_delta) = 1
```

In CLVM:
```clojure
(bls_pairing_identity
    A VK_BETA
    (g1_negate VK_ALPHA) B
    (g1_negate vk_input) VK_GAMMA
    (g1_negate C) VK_DELTA)
```

Where `vk_input` is computed from public inputs.

### VK Input Computation

```clojure
vk_input = IC[0]
         + scalar(validator_merkle_root) × IC[1]
         + scalar(validator_count) × IC[2]
         + scalar(new_validator_merkle_root) × IC[3]
         + scalar(new_validator_count) × IC[4]
         + scalar(agg_signers) × IC[5]
         + scalar(checkpoint_message) × IC[6]
```

Using `g1_multiply` and `g1_add`:
```clojure
(defun compute_vk_input (ic inputs)
    (if (not inputs)
        (first ic)  ; IC[0]
        (g1_add
            (g1_multiply (first ic) (scalar (first inputs)))
            (compute_vk_input (rest ic) (rest inputs)))))
```

### scalar() Function

```
scalar(bytes) = sha256(bytes) interpreted as big-endian u256, reduced mod r
```

Where `r` is the BLS12-381 scalar field order.

### BLS Signature Verification

After Groth16 passes, verify the aggregate signature:

```clojure
(bls_verify agg_sig agg_signers checkpoint_message)
```

Where:
- `agg_sig`: G2 point (96 bytes) — aggregate of k individual signatures
- `agg_signers`: G1 point (48 bytes) — aggregate of k signing pubkeys
- `checkpoint_message`: 32 bytes — hash of new state

### Checkpoint Message Format

```
checkpoint_message = sha256(
    new_state_root              // 32 bytes
    + new_validator_merkle_root // 32 bytes
    + new_validator_count       // 8 bytes, big-endian u64
    + new_epoch                 // 8 bytes, big-endian u64
)
```

### Public Inputs to Circuit

| Index | Input | Source |
|-------|-------|--------|
| 1 | `validator_merkle_root` | Current singleton state |
| 2 | `validator_count` | Current singleton state |
| 3 | `new_validator_merkle_root` | Solution parameter |
| 4 | `new_validator_count` | Solution parameter |
| 5 | `agg_signers` | Solution parameter |
| 6 | `checkpoint_message` | Computed from solution |

### What the Circuit Proves

> "I know k BLS pubkeys, each with a valid Merkle inclusion proof against `validator_merkle_root`, whose G1 sum equals `agg_signers`, and where 2k > `validator_count`."

The circuit does NOT verify the BLS signature — that's `bls_verify`'s job.

### CLVM Cost

| Operation | Cost |
|-----------|------|
| `bls_pairing_identity` (4 pairs) | ~7,800,000 |
| `bls_verify` | ~4,200,000 |
| VK input computation | ~707,000 |
| Other puzzle logic | ~4,500,000 |
| **Total checkpoint spend** | **~17,229,000** |

This is 0.157% of the 11B block limit.

## Acceptance Criteria

- [ ] Groth16 proof verified via `bls_pairing_identity`
- [ ] VK input computed from 6 public inputs
- [ ] scalar() uses sha256 mod r
- [ ] BLS signature verified via `bls_verify`
- [ ] Valid proof + valid signature → spend succeeds
- [ ] Invalid proof → spend fails
- [ ] Valid proof + invalid signature → spend fails
- [ ] Minority signers (2k ≤ count) → proof generation fails

## Implementation Notes

- **Puzzle source:** `puzzles/checkpoint_inner.clsp`
- **Verification key:** Curried in, 672 bytes
- **Proof:** Passed in solution, 192 bytes

### Verification Key Format

```
VK = alpha_g1 (48) + beta_g2 (96) + gamma_g2 (96) + delta_g2 (96) + 7×ic (336)
   = 672 bytes total
```

## Verification

1. Generate valid checkpoint (k signers, valid proof)
2. Submit → verify success
3. Corrupt proof.a by 1 byte → verify failure
4. Restore proof, corrupt signature → verify failure
5. Reduce signers to minority → verify proof generation fails
6. Use pubkeys not in Merkle tree → verify proof generation fails
