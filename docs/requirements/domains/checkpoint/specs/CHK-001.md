# CHK-001 — Checkpoint Singleton Identity

> **Authoritative requirement:** [CHK-001](../NORMATIVE.md#CHK-001)
> **Verification:** [VERIFICATION.md](../VERIFICATION.md)
> **Tracking:** [TRACKING.yaml](../TRACKING.yaml)
> **CHIP reference:** [chip-groth16-l2-consensus.md](../../../../resources/chip-groth16-l2-consensus.md) — Part 3: Checkpoint Singleton

## Summary

The checkpoint singleton is the canonical on-chain source of truth for L2 state. It MUST be implemented as a Chia singleton to guarantee exactly one authoritative state exists. All L2 operations (checkpoints, membership queries, collateral recovery) depend on this singleton as the single source of truth.

## Specification

### Singleton Structure

```
outer_puzzle = singleton_top_layer_v1_1(
    SINGLETON_STRUCT,        # (MOD_HASH, LAUNCHER_ID, LAUNCHER_PUZZLE_HASH)
    inner_puzzle             # checkpoint_inner.clsp
)
```

### Role in the System

The checkpoint singleton serves as:

| Function | How |
|----------|-----|
| State authority | Curried state values are the official L2 state |
| Checkpoint verifier | Validates Groth16 proofs and BLS signatures |
| Membership oracle | Emits announcements for collateral recovery |
| Epoch tracker | Maintains monotonic epoch counter |

### Singleton Properties

1. **Uniqueness**: Exactly one checkpoint singleton exists per L2 network
2. **Persistence**: Always recreates itself (never consumed)
3. **Lineage**: Full history traceable back to launcher
4. **Immutable binding**: VK and system parameters curried at deployment

### Deployment Relationship

```
Genesis Coin
    │
    ├─► Network Coin Launcher ─► Network Coin
    │
    └─► Checkpoint Launcher ─► Checkpoint Singleton
```

Both singletons share a relationship through the genesis coin but maintain independent lineages.

### Identification

- **Launcher ID**: Permanent identifier (used in configurations)
- **Coin ID**: Current instance identifier (changes each spend, used in announcements)
- **Puzzle Hash**: Varies with state (curried values change)

### Why Singleton Matters

Without singleton enforcement:
1. Multiple "checkpoint singletons" could exist
2. Validators could submit checkpoints to different singletons
3. No single authoritative state would exist
4. Registration coins couldn't reliably assert announcements

The singleton wrapper guarantees that regardless of how many checkpoints occur, there is always exactly one checkpoint singleton with the current state.

## Acceptance Criteria

- [ ] Checkpoint singleton uses `singleton_top_layer_v1_1` wrapper
- [ ] Launcher ID is derived from genesis coin spend
- [ ] Only one checkpoint singleton can exist at any time
- [ ] Both spend paths (checkpoint and membership query) recreate singleton
- [ ] Singleton lineage is verifiable from any spend
- [ ] Launcher ID is stable and documented in network config

## Implementation Notes

- **Puzzle source:** `puzzles/checkpoint_inner.rue` (Rue)
- **Compiled hex:** `puzzles/compiled/checkpoint_inner.hex` (CLVM bytecode)
- **Compiled hash:** `puzzles/compiled/checkpoint_inner.hash` (tree hash)
- **Driver code:** `src/puzzles/checkpoint.rs`
- **Dependencies:** Chia singleton standard

### Compile Pipeline

The Rue source is compiled to `.hex` (CLVM bytecode) and `.hash` (tree hash),
which are embedded into the Rust driver via `include_str!`:

```
puzzles/checkpoint_inner.rue
   │  rue build -x  → puzzles/compiled/checkpoint_inner.hex
   │  rue build --hash → puzzles/compiled/checkpoint_inner.hash
   ▼
src/puzzles/checkpoint.rs
   CHECKPOINT_INNER_PUZZLE_HEX  = include_str!("...hex")
   CHECKPOINT_INNER_MOD_HASH_HEX = include_str!("...hash")
```

**Note:** checkpoint_inner.rue does not yet compile — it requires Rue syntax
fixes (Phase 3). The compile pipeline pattern is the same as registration_coin
and network_coin_inner.

### Co-deployment with Network Coin

For proper operation, both singletons must know each other's launcher IDs:
- Checkpoint needs to know its own ID (for state tracking)
- Network coin needs checkpoint launcher ID (for registration coins)
- Both can be derived from a shared genesis coin

## Verification

1. Deploy checkpoint singleton via genesis spend
2. Verify singleton wrapper present in outer puzzle
3. Query blockchain for coins with checkpoint puzzle hash prefix
4. Verify exactly one coin exists at any time
5. Submit checkpoint → verify new singleton created
6. Verify lineage: new coin's parent is previous singleton
7. Attempt to create duplicate → verify failure
