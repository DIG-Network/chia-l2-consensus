# CHK-004 — State Update and Announcement

> **Authoritative requirement:** [CHK-004](../NORMATIVE.md#CHK-004)
> **Verification:** [VERIFICATION.md](../VERIFICATION.md)
> **Tracking:** [TRACKING.yaml](../TRACKING.yaml)
> **CHIP reference:** [chip-groth16-l2-consensus.md](../../../../resources/chip-groth16-l2-consensus.md) — Part 3: Checkpoint Singleton

## Summary

After successful Groth16 and BLS verification, the checkpoint singleton updates its state, emits a checkpoint announcement, and recreates itself with the new state. The epoch increment is automatic and enforced by the puzzle.

## Specification

### State Update Rules

| Field | Update Rule |
|-------|-------------|
| `state_root` | Set to `new_state_root` from solution |
| `epoch` | Set to `old_epoch + 1` (automatic) |
| `validator_merkle_root` | Set to `new_validator_merkle_root` from solution |
| `validator_count` | Set to `new_validator_count` from solution |

### Epoch Increment

The epoch increment is hardcoded in the puzzle:

```clojure
(defun compute_new_epoch (current_epoch)
    (+ current_epoch 1))

; Used when recreating singleton
new_epoch = compute_new_epoch(EPOCH)
```

The solution cannot specify the new epoch — it's always current + 1.

### Singleton Recreation

After verification, the puzzle creates the successor singleton:

```clojure
(CREATE_COIN
    (curry_hash
        CHECKPOINT_INNER_MOD_HASH
        new_state_root
        new_epoch                    ; old + 1
        new_validator_merkle_root
        new_validator_count
        VK TREE_DEPTH EMPTY_LEAF_HASH)
    MY_AMOUNT)                       ; 1 mojo
```

### Checkpoint State Announcement

The puzzle emits an announcement for L2 observers:

```clojure
(CREATE_COIN_ANNOUNCEMENT
    (sha256
        "checkpoint"                 ; 10 bytes
        new_epoch_be8                ; 8 bytes
        new_state_root               ; 32 bytes
        new_validator_merkle_root    ; 32 bytes
        new_validator_count_be8))    ; 8 bytes
```

This announcement allows:
- L2 nodes to observe checkpoint acceptance
- External systems to verify state transitions
- Audit trails of consensus history

### Complete Checkpoint Output

A successful checkpoint spend produces:

```clojure
(list
    ; Groth16 verification (implicit - spend fails if invalid)
    ; BLS verification (implicit - spend fails if invalid)

    ; State announcement
    (CREATE_COIN_ANNOUNCEMENT checkpoint_announcement)

    ; Singleton recreation with new state
    (CREATE_COIN new_singleton_puzzle_hash 1)
)
```

### Atomicity

The entire checkpoint is atomic:
- Either all verifications pass AND state updates AND singleton recreates
- Or the entire spend fails, leaving previous state unchanged

No partial checkpoints are possible.

### Solution Parameters

```clojure
; Checkpoint spend path solution
spend_path = 0  ; Indicates checkpoint (vs 1 for membership query)
new_state_root
new_validator_merkle_root
new_validator_count
agg_signers
agg_sig
proof_a
proof_b
proof_c
```

### Validation of New Values

The puzzle does NOT independently validate that:
- `new_state_root` is "correct" (application-defined)
- `new_validator_merkle_root` matches actual registration coins
- `new_validator_count` is accurate

These are validated by the majority signature over the checkpoint message. A majority of validators attest that these values are correct by signing.

## Acceptance Criteria

- [ ] Epoch increments by exactly 1
- [ ] state_root updated to solution value
- [ ] validator_merkle_root updated to solution value
- [ ] validator_count updated to solution value
- [ ] Checkpoint announcement emitted with correct format
- [ ] New singleton created with updated state
- [ ] Old singleton spent (no longer exists)
- [ ] Epoch cannot be manipulated by solution

## Implementation Notes

- **Primary codebase:** `puzzles/checkpoint_inner.clsp`
- **Driver code:** `src/checkpoint.rs` — `build_checkpoint_spend()`

### Reading New State

After checkpoint, query the new singleton:

```rust
// Before: epoch = 5
client.sync();
assert_eq!(client.epoch(), 5);

// Submit checkpoint
let bundle = client.submit_checkpoint(...);
submit_and_wait(bundle);

// After: epoch = 6
client.sync();
assert_eq!(client.epoch(), 6);
```

### Announcement Observation

L2 nodes can watch for checkpoint announcements:

```rust
fn on_new_coin_announcement(announcement: &Announcement) {
    if announcement.starts_with(b"checkpoint") {
        let (epoch, state_root, merkle_root, count) = parse_checkpoint_announcement(announcement);
        println!("New checkpoint at epoch {}", epoch);
    }
}
```

## Verification

1. Query current checkpoint state (epoch N)
2. Build and submit valid checkpoint
3. Query new checkpoint state
4. Verify epoch = N + 1
5. Verify state values match solution
6. Verify old singleton no longer exists
7. Verify new singleton exists with new puzzle hash
8. Verify announcement emitted with correct values
