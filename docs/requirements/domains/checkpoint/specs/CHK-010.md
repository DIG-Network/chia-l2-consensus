# CHK-010 — Single Checkpoint Per Epoch

> **Authoritative requirement:** [CHK-010](../NORMATIVE.md#CHK-010)
> **Verification:** [VERIFICATION.md](../VERIFICATION.md)
> **Tracking:** [TRACKING.yaml](../TRACKING.yaml)
> **CHIP reference:** [chip-groth16-l2-consensus.md](../../../../resources/chip-groth16-l2-consensus.md) — Part 3: Checkpoint Singleton

## Summary

Only one checkpoint can be accepted per epoch. Three independent mechanisms
enforce this: the singleton coin model (spending destroys the old coin),
the epoch in the checkpoint_message (binding proof to epoch), and the BLS
signature (validators sign a message containing the epoch).

## Specification

### Mechanism 1: Singleton Pattern

The checkpoint singleton is a Chia coin. Spending it consumes the coin
and creates a new coin with epoch+1. The old coin no longer exists and
cannot be spent again. This is the primary enforcement.

```
Epoch 5: Coin A exists (epoch=5 curried in)
    ↓  Checkpoint spend consumes Coin A
Epoch 6: Coin B exists (epoch=6 curried in)
         Coin A is SPENT — cannot be used again
```

### Mechanism 2: Epoch in Checkpoint Message

Even if an attacker could somehow recreate an old coin, the epoch in the
checkpoint_message prevents replay:

```
Proof at epoch 5 → checkpoint_message = sha256(...‖epoch=6‖...)
Proof at epoch 9 → checkpoint_message = sha256(...‖epoch=10‖...)
```

Different epoch → different message → different scalar → spend fails.

### Mechanism 3: BLS Signature Binding

Validators sign the full checkpoint_message which includes the epoch.
A signature for epoch 6 is invalid for epoch 10.

### Race Condition: Two Proofs for Same Epoch

If two submitters both generate proofs for epoch N+1 simultaneously:
1. Both proofs are valid (both bind to epoch N+1)
2. First submitted proof spends the coin → epoch becomes N+1
3. Second proof tries to spend — but the coin is already SPENT
4. Second proof's spend is rejected (coin doesn't exist)

This is the standard Chia UTXO model — first-writer-wins.

## Acceptance Criteria

- [ ] Spending checkpoint singleton consumes the old coin
- [ ] New coin has epoch = old_epoch + 1
- [ ] Attempting to spend an already-spent coin fails
- [ ] Two checkpoint spends at the same epoch: only first succeeds
- [ ] Proof for epoch N cannot verify when singleton is at epoch M ≠ N-1
- [ ] BLS signature for epoch N does not verify at epoch M
