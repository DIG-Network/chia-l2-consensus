# CHK-012 — Network ID Binding

> **Authoritative requirement:** [CHK-012](../NORMATIVE.md#CHK-012)
> **Verification:** [VERIFICATION.md](../VERIFICATION.md)
> **Tracking:** [TRACKING.yaml](../TRACKING.yaml)
> **CHIP reference:** [chip-groth16-l2-consensus.md](../../../../resources/chip-groth16-l2-consensus.md) — Security

## Summary

Including the network_coin_launcher_id in the checkpoint_message prevents
cross-network proof replay. Without it, if two L2 networks share the same
trusted setup and circuit parameters, a proof from network A could
theoretically be replayed on network B.

## Specification

### The Attack Without Network Binding

1. Network A and B both use the same circuit (same MAX_SIGNERS, TREE_DEPTH)
2. Network A has validators {V1, V2, V3}; Network B has the same set
3. Attacker generates a valid proof+signature on Network A
4. Attacker submits the same proof+signature to Network B
5. Without network ID in the message, both networks accept it

### The Fix

Add `network_coin_launcher_id` to the checkpoint_message preimage:

```
checkpoint_message = sha256(
    new_state_root              (32 bytes)
    + new_validator_merkle_root (32 bytes)
    + new_validator_count       (8 bytes, u64 BE)
    + new_epoch                 (8 bytes, u64 BE)
    + network_coin_launcher_id  (32 bytes)    ← NEW
)
```

Total preimage: 112 bytes (was 80 bytes).

### Implementation Changes Required

1. **Rue puzzle (checkpoint_inner.rue)**:
   - Add `NETWORK_COIN_LAUNCHER_ID` as a curried parameter
   - Include it in the checkpoint_message sha256 computation
   - Curried at deployment, not from solution (cannot be spoofed)

2. **Rust wire format (src/prover/serialize.rs)**:
   - Update `compute_checkpoint_message()` to accept and include network_coin_launcher_id
   - Update message size constant from 80 to 112

3. **Validator signing (src/validator/signing.rs)**:
   - Update `compute_checkpoint_signing_message()` — no change needed
     (it uses the output of compute_checkpoint_message, which now includes network ID)

4. **Circuit**:
   - No change needed — circuit receives checkpoint_message as a 32-byte hash
   - The hash now includes network ID, so the proof is automatically bound

### Why Curried (Not From Solution)

The network_coin_launcher_id MUST be curried into the puzzle at deployment
because:
- If from solution, attacker can supply any network ID
- Curried values are part of the puzzle hash, committed on-chain
- Cannot be changed without redeploying the singleton

### Existing Partial Binding

Currently, the validator signing message includes:
```
signing_msg = checkpoint_message + genesis_challenge + checkpoint_coin_id
```

- `genesis_challenge` binds to Chia mainnet/testnet (not to a specific L2)
- `checkpoint_coin_id` changes per spend (provides some uniqueness)
- Neither explicitly identifies the L2 network

The `network_coin_launcher_id` provides the missing L2 network identity.

## Acceptance Criteria

- [ ] checkpoint_message includes network_coin_launcher_id in sha256 preimage
- [ ] network_coin_launcher_id is curried into checkpoint puzzle (not from solution)
- [ ] Different network IDs produce different checkpoint_messages
- [ ] Proof from network A fails on network B (different message)
- [ ] compute_checkpoint_message() accepts network_coin_launcher_id parameter
- [ ] Rue puzzle includes network_coin_launcher_id in message computation
- [ ] Total preimage is 112 bytes (32+32+8+8+32)
