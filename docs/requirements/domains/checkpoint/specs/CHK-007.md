# CHK-007 — Verification Key Immutability

> **Authoritative requirement:** [CHK-007](../NORMATIVE.md#CHK-007)
> **Verification:** [VERIFICATION.md](../VERIFICATION.md)
> **Tracking:** [TRACKING.yaml](../TRACKING.yaml)
> **CHIP reference:** [chip-groth16-l2-consensus.md](../../../../resources/chip-groth16-l2-consensus.md) — Part 3: Checkpoint Singleton

## Summary

The Groth16 verification key is curried into the checkpoint singleton at deployment and cannot be changed. This permanently binds the singleton to a specific circuit and trusted setup, ensuring all proofs are verified against the same cryptographic parameters.

## Specification

### Verification Key Structure

| Component | Size | Description |
|-----------|------|-------------|
| `alpha_g1` | 48 bytes | G1 point from trusted setup |
| `beta_g2` | 96 bytes | G2 point from trusted setup |
| `gamma_g2` | 96 bytes | G2 point from trusted setup |
| `delta_g2` | 96 bytes | G2 point from trusted setup |
| `ic[0..6]` | 336 bytes | 7 G1 points for public inputs |
| **Total** | **672 bytes** | |

### VK in Puzzle Structure

```clojure
(mod (
    ; State (changes each checkpoint)
    STATE_ROOT EPOCH VALIDATOR_MERKLE_ROOT VALIDATOR_COUNT

    ; Constants (never change)
    VK                      ; 672 bytes, curried at deployment
    TREE_DEPTH              ; Integer, matches circuit
    EMPTY_LEAF_HASH         ; 32 bytes, precomputed

    ; Solution
    ...
)
```

### Immutability Mechanism

The VK is curried, not passed in solution:
- Curried values are part of the puzzle hash
- Changing VK would change the puzzle hash
- Changing puzzle hash would break singleton lineage
- Therefore VK cannot change without "breaking" the singleton

### Why 7 IC Points

| IC Index | Public Input |
|----------|-------------|
| 0 | Constant term (no multiplication) |
| 1 | validator_merkle_root |
| 2 | validator_count |
| 3 | new_validator_merkle_root |
| 4 | new_validator_count |
| 5 | agg_signers |
| 6 | checkpoint_message |

The circuit has 6 public inputs, plus 1 constant term = 7 IC points.

### VK Binding Properties

The VK binds the checkpoint singleton to:

1. **Specific circuit**: Different circuits have different VKs
2. **Specific trusted setup**: Each ceremony produces unique VK
3. **Specific parameters**: MAX_SIGNERS, TREE_DEPTH baked into VK

### Deployment-Time VK Verification

Before deployment, operators MUST verify:

```bash
# 1. VK was produced by legitimate ceremony
$ sha256sum verification_key.bin
abc123...  # Compare to published ceremony output

# 2. VK has correct IC count
$ cargo run --bin vk-inspect verification_key.bin
IC points: 7 ✓

# 3. VK matches circuit parameters
$ cargo run --bin circuit-check --max-signers 64 --tree-depth 32
Circuit parameters match VK ✓
```

### What Happens If VK Is Wrong

| Scenario | Effect |
|----------|--------|
| VK from wrong ceremony | Valid-looking proofs from compromised setup |
| VK from wrong circuit | All proofs fail (circuit mismatch) |
| VK with wrong IC count | All proofs fail (public input mismatch) |
| VK with wrong parameters | Proofs fail or security violated |

### Updating the VK

If the VK needs to change (e.g., after security incident):
1. Run new trusted setup ceremony
2. Deploy NEW checkpoint singleton with new VK
3. Migrate validators to new singleton
4. Old singleton becomes defunct

This is intentionally difficult — VK changes are a "nuclear option."

### VK Publication

The VK MUST be published for independent verification:
- In network configuration file
- In deployment documentation
- In ceremony transcript

Anyone can verify the deployed singleton contains the expected VK by decurrying.

## Acceptance Criteria

- [ ] VK curried into checkpoint puzzle at deployment
- [ ] VK is exactly 672 bytes
- [ ] VK contains 7 IC points (G1)
- [ ] VK contains alpha_g1, beta_g2, gamma_g2, delta_g2
- [ ] VK cannot be changed via solution parameters
- [ ] Decurrying reveals correct VK
- [ ] VK matches published ceremony output

### Serialization Format

```rust
struct VerificationKey {
    alpha_g1: G1Affine,      // 48 bytes compressed
    beta_g2: G2Affine,       // 96 bytes compressed
    gamma_g2: G2Affine,      // 96 bytes compressed
    delta_g2: G2Affine,      // 96 bytes compressed
    ic: Vec<G1Affine>,       // 7 × 48 = 336 bytes
}

fn serialize(&self) -> [u8; 672] {
    // ZCash compressed format for all points
}
```

## Verification

1. Deploy checkpoint singleton
2. Decurry to extract VK
3. Verify VK size = 672 bytes
4. Verify IC count = 7
5. Compare VK to ceremony output
6. Submit checkpoint → verify VK used in verification
7. Attempt to submit checkpoint with different VK embedded → impossible
