# REG-007 — End-to-End Simulator Test

> **Authoritative requirement:** [REG-007](../NORMATIVE.md#REG-007)
> **Verification:** [VERIFICATION.md](../VERIFICATION.md)
> **Tracking:** [TRACKING.yaml](../TRACKING.yaml)
> **CHIP reference:** [chip-groth16-l2-consensus.md](../../../../resources/chip-groth16-l2-consensus.md) — Part 2: Registration Coin, De-Registration and Collateral Recovery

## Summary

All prior REG requirements (REG-001 through REG-006) were verified using CLVM execution tests that run the puzzle in isolation. This requirement mandates a full end-to-end simulator test that exercises the registration coin through its complete lifecycle — creation via network coin, collateral lock enforcement, and collateral recovery via cross-coin announcement matching between the checkpoint singleton and registration coin in the same spend bundle.

This is the test that proves the cross-coin announcement protocol actually works: the checkpoint singleton emits a non-membership announcement, the registration coin asserts that exact announcement, and the simulator validates the match at the consensus level.

## Specification

### Test Lifecycle

```
1. SETUP
   - Deploy network coin singleton via chia-wallet-sdk
   - Deploy checkpoint singleton via chia-wallet-sdk
   - Both singletons live in the same simulator

2. CREATE REGISTRATION COIN
   - Register validator by spending network coin (see NET-006)
   - Verify: registration coin exists with:
     * Parent = network coin spend
     * Puzzle hash = curry_hash(REG_MOD_HASH, pk, checkpoint_coin_id)
     * Amount = COLLATERAL_AMOUNT
   - Verify: registration coin is NOT spendable yet (no announcement available)

3. COLLATERAL LOCK ENFORCEMENT
   - Attempt to spend registration coin without any announcement
   - Verify: simulator rejects the spend (ASSERT_COIN_ANNOUNCEMENT fails)
   - Attempt to spend registration coin with a fabricated announcement
     (correct format but from wrong coin ID)
   - Verify: simulator rejects (announcement source mismatch)

4. COLLATERAL RECOVERY (the cross-coin test)
   Build a single spend bundle containing two spends:

   Spend 1: Checkpoint Singleton (membership query path)
     - Solution: query_pubkey, is_member=false, merkle_siblings
     - Provides a valid non-membership Merkle proof
     - Output: CREATE_COIN_ANNOUNCEMENT with non-membership hash
     - Output: CREATE_COIN to recreate singleton unchanged

   Spend 2: Registration Coin
     - Solution: epoch, collateral_destination, collateral_amount, []
     - Output: ASSERT_COIN_ANNOUNCEMENT matching Spend 1's announcement
     - Output: CREATE_COIN to return collateral

   Submit this bundle to the simulator.
   Verify: both spends succeed atomically
   Verify: registration coin is spent
   Verify: collateral coin exists at destination with correct amount
   Verify: checkpoint singleton recreated unchanged (same state)

5. EPOCH REPLAY ATTACK
   - Re-register the same validator (new registration coin)
   - Advance to a new checkpoint that includes them
   - Attempt collateral recovery using the old epoch
   - Verify: simulator rejects (announcement hash mismatch)

6. FAILURE CASES
   - Spend registration coin with is_member=0x01 → rejected
   - Spend registration coin asserting wrong checkpoint coin ID → rejected
   - Spend registration coin with wrong epoch → rejected
   - Spend registration coin without checkpoint query in bundle → rejected
```

### Cross-Coin Announcement Matching

This is the critical test. The announcement protocol:

```
Checkpoint emits:
  CREATE_COIN_ANNOUNCEMENT(
    sha256("membership" + epoch_be8 + pubkey + 0x00)
  )

Registration coin asserts:
  ASSERT_COIN_ANNOUNCEMENT(
    sha256(checkpoint_coin_id + sha256("membership" + epoch_be8 + pubkey + 0x00))
  )
```

The simulator validates:
1. The announcing coin (checkpoint) actually exists and is being spent
2. The announcement content matches what the asserting coin expects
3. The announcing coin's ID matches what's embedded in the assertion hash

### What This Test Proves Beyond REG-001..006

| Prior test | What it proved | What it missed |
|-----------|---------------|----------------|
| REG-001 (CLVM) | Puzzle has 2 curried params | Params work with real checkpoint singleton |

| REG-003 (CLVM) | ASSERT_COIN_ANNOUNCEMENT emitted | Announcement actually matched by checkpoint |
| REG-004 (CLVM) | 67-byte preimage format correct | Hash matches checkpoint's actual output |
| REG-005 (CLVM) | CREATE_COIN from solution values | Collateral coin actually created on-chain |
| REG-006 (CLVM) | Different epochs → different hashes | Epoch mismatch actually rejected by simulator |
| **REG-007 (simulator)** | **All of the above in a real spend bundle** | **N/A — definitive test** |

### Specific Assertions

After successful collateral recovery:

```rust
// Registration coin spent
assert!(sim.coin_state(registration_coin_id).is_spent());

// Collateral returned
let collateral_coins = sim.coin_records_by_puzzle_hash(destination_puzzle_hash);
assert_eq!(collateral_coins.len(), 1);
assert_eq!(collateral_coins[0].coin.amount, COLLATERAL_AMOUNT);

// Checkpoint singleton unchanged
let checkpoint_coins = sim.coin_records_by_puzzle_hash(checkpoint_puzzle_hash);
assert_eq!(checkpoint_coins.len(), 1);
// Decurry and verify state unchanged (same epoch, same merkle root)
```

### Relationship to CHK-008

CHK-008 is the full system E2E test including Groth16 proof generation. REG-007 focuses specifically on the registration coin lifecycle and cross-coin announcement protocol. REG-007 can use a pre-populated checkpoint singleton state (curried with known Merkle root and epoch) without needing the Groth16 prover, making it faster to run.

| Test | Scope | Groth16 needed | Duration |
|------|-------|---------------|----------|
| REG-007 | Registration + collateral recovery | No | < 5 seconds |
| CHK-008 | Full lifecycle including proof generation | Yes | 5-15 minutes |

## Acceptance Criteria

- [ ] Registration coin created via network coin spend in simulator
- [ ] Collateral lock enforced: spend without announcement rejected
- [ ] Cross-coin bundle: checkpoint query + registration coin spend succeeds
- [ ] Announcement hashes match between checkpoint and registration coin
- [ ] Collateral returned to correct destination with correct amount
- [ ] Checkpoint singleton state unchanged after membership query
- [ ] Epoch replay attack rejected by simulator
- [ ] Wrong checkpoint coin ID rejected by simulator
- [ ] Fabricated announcement (wrong source) rejected by simulator
- [ ] All assertions use simulator state, not CLVM output parsing

### Building the Cross-Coin Bundle

```rust
let mut ctx = SpendContext::new();

// Spend 1: Checkpoint membership query
let checkpoint_spend = build_membership_query(
    &checkpoint_coin,
    &validator_pubkey,
    false,  // is_member = false (non-membership)
    &non_membership_proof,
    current_epoch,
);
ctx.insert(checkpoint_spend);

// Spend 2: Registration coin recovery
let reg_spend = build_collateral_recovery(
    &registration_coin,
    current_epoch,
    destination_puzzle_hash,
    COLLATERAL_AMOUNT,
);
ctx.insert(reg_spend);

// Submit both as one bundle
sim.spend_coins(ctx.take(), &[])?;
```
