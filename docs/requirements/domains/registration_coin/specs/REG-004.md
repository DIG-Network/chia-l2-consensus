## Summary

The registration coin enforces collateral lock by asserting a specific announcement from the checkpoint singleton. The announcement format is precisely defined to prevent manipulation and ensure only the correct checkpoint singleton can authorize collateral recovery.

## Specification

### Announcement Format

The registration coin asserts:

```
ASSERT_COIN_ANNOUNCEMENT(full_announcement_hash)
```

Where:
```
full_announcement_hash = sha256(
    CHECKPOINT_SINGLETON_ID           # 32 bytes - the announcing coin
    + inner_announcement              # 32 bytes - the announcement content
)

inner_announcement = sha256(
    "membership"                      # 10 bytes UTF-8, no null
    + epoch                           # 8 bytes big-endian u64
    + VALIDATOR_PUBKEY                # 48 bytes
    + is_member                       # 1 byte: 0x00=not member, 0x01=member
)
```

### Byte Layout

| Field | Offset | Size | Format | Description |
|-------|--------|------|--------|-------------|
| `"membership"` | 0 | 10 | UTF-8 | Literal string prefix |
| `epoch` | 10 | 8 | u64 BE | Current checkpoint epoch |
| `pubkey` | 18 | 48 | G1 compressed | Validator's BLS pubkey |
| `is_member` | 66 | 1 | byte | 0x00 for non-member |

Total inner content: 67 bytes → hashed to 32 bytes

### Why This Format

**String prefix "membership":**
- Prevents collision with other announcement types
- Human-readable in debugging
- Fixed length (10 bytes)

**Epoch inclusion:**
- Replay protection (see [REG-006](REG-006.md))
- Ties announcement to specific checkpoint state

**Pubkey inclusion:**
- Ties announcement to specific validator
- Prevents using another validator's announcement

**is_member byte:**
- Distinguishes membership from non-membership queries
- Registration coin requires `0x00` (non-member)

### Puzzle Implementation

```clojure
(mod (
    VALIDATOR_PUBKEY
    CHECKPOINT_SINGLETON_ID

    epoch
    collateral_destination
    collateral_amount
    conditions
)
    ; Compute expected announcement
    (defun compute_inner_announcement (epoch pubkey)
        (sha256
            "membership"                    ; 10 bytes
            (int_to_bytes epoch 8)          ; 8 bytes BE
            pubkey                          ; 48 bytes
            0x00                            ; 1 byte - non-member
        )
    )

    (defun compute_full_announcement (singleton_id inner)
        (sha256 singleton_id inner)
    )

    (c
        (list ASSERT_COIN_ANNOUNCEMENT
            (compute_full_announcement
                CHECKPOINT_SINGLETON_ID
                (compute_inner_announcement epoch VALIDATOR_PUBKEY)
            )
        )
        ; ... rest of conditions
    )
)
```

### Matching Checkpoint Announcement

The checkpoint singleton membership query emits:

```clojure
(CREATE_COIN_ANNOUNCEMENT
    (sha256 "membership" epoch_be8 query_pubkey is_member))
```

For the registration coin to succeed:
1. Checkpoint announces for this exact pubkey
2. Checkpoint announces is_member=0x00
3. Checkpoint announces current epoch
4. Registration coin asserts against checkpoint's coin ID

### Why CHECKPOINT_SINGLETON_ID is Coin ID

The curried `CHECKPOINT_SINGLETON_ID` is the coin ID (not launcher ID) because:
- `ASSERT_COIN_ANNOUNCEMENT` requires the announcing coin's ID
- Launcher ID doesn't identify a specific coin instance
- The coin ID changes with each checkpoint spend

This creates a coupling: registration coins are bound to a specific checkpoint singleton coin. When checkpoint advances, old registration coins may need the new coin ID. This is handled by the driver.

## Acceptance Criteria

- [ ] Announcement hash computed exactly per specification
- [ ] String "membership" is 10 bytes UTF-8
- [ ] Epoch is 8-byte big-endian u64
- [ ] Pubkey is 48-byte compressed G1
- [ ] is_member is single byte 0x00
- [ ] Full announcement includes checkpoint singleton coin ID
- [ ] Spend fails if any byte differs from expected

## Implementation Notes

- **Primary codebase:** `puzzles/registration_coin.clsp`
- **Driver code:** `src/registration_coin.rs` — `compute_announcement_hash()`
- **Cross-impl test:** Rust hash must match Chialisp hash exactly

### Common Mistakes

1. Using launcher ID instead of coin ID
2. Wrong byte order for epoch (must be big-endian)
3. Missing or wrong string prefix
4. is_member = 0x01 instead of 0x00

## Verification

1. Compute inner announcement hash manually
2. Compute full announcement hash manually
3. Compare against puzzle output
4. Submit spend with correct announcement → success
5. Submit spend with wrong epoch → failure
6. Submit spend with wrong pubkey → failure
7. Submit spend with is_member=0x01 → failure
8. Submit spend with wrong singleton ID → failure
