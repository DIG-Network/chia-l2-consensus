# REG-009 — Failure Case Coverage

> **Authoritative requirement:** [REG-009](../NORMATIVE.md#REG-009)
> **Verification:** [VERIFICATION.md](../VERIFICATION.md)
> **Tracking:** [TRACKING.yaml](../TRACKING.yaml)

## Summary

REG-007 has 2 simulator tests but all are success-path only. Per SCHEMA.md
Permutation Testing requirements, tests MUST cover failure cases. This
requirement adds negative-path coverage to prove the collateral lock works.

## Specification

### Test 1: Spend without announcement
- Build registration coin spend bundle with NO checkpoint singleton spend
- Submit to simulator → must be rejected (ASSERT_COIN_ANNOUNCEMENT unsatisfied)

### Test 2: Wrong announcement hash
- Build cross-coin bundle but corrupt the announcement hash
- Submit to simulator → must be rejected

### Test 3: Membership=true announcement
- Build cross-coin bundle where checkpoint announces is_member=0x01 (still active)
- Registration coin expects is_member=0x00
- Submit to simulator → must be rejected

### Test 4: Wrong epoch
- Build cross-coin bundle with epoch N in registration coin solution
  but checkpoint is at epoch M (N ≠ M)
- Submit to simulator → must be rejected

## Acceptance Criteria

- [x] Spend without announcement is rejected by simulator — `vv_req_reg_009_spend_without_announcement`
- [x] Spend with wrong announcement hash is rejected — `vv_req_reg_009_wrong_announcement_hash`
- [x] Spend with is_member=0x01 is rejected — `vv_req_reg_009_membership_true_rejected`
- [x] Spend with wrong epoch is rejected — `vv_req_reg_009_wrong_epoch_rejected`
