# REG-005 — Collateral Return

> **Authoritative requirement:** [REG-005](../NORMATIVE.md#REG-005)
> **Verification:** [VERIFICATION.md](../VERIFICATION.md)
> **Tracking:** [TRACKING.yaml](../TRACKING.yaml)
> **CHIP reference:** [chip-groth16-l2-consensus.md](../../../../resources/chip-groth16-l2-consensus.md) — Part 2: Registration Coin

## Summary

Upon valid spend (with non-membership announcement), the registration coin creates a new coin at the validator-specified destination with the full collateral amount. This returns the locked funds to the exiting validator.

## Specification

### Output Condition

The registration coin emits:

```clojure
(CREATE_COIN collateral_destination collateral_amount)
```

Where:
- `collateral_destination`: Puzzle hash specified in the solution (32 bytes)
- `collateral_amount`: Amount specified in the solution (mojos)

### Solution Parameters

| Parameter | Type | Description |
|-----------|------|-------------|
| `epoch` | u64 | Current checkpoint epoch (for announcement) |
| `collateral_destination` | Bytes32 | Puzzle hash to receive collateral |
| `collateral_amount` | u64 | Amount to send (should equal coin amount) |
| `conditions` | List | Additional conditions (optional) |

### Destination Flexibility

The validator chooses the destination puzzle hash:
- Standard wallet address (p2_puzzle_hash)
- Multi-sig wallet
- Another smart contract
- Any valid puzzle hash

This flexibility allows validators to:
- Send to a different wallet than registration source
- Implement custom fund management
- Use hardware wallet addresses

### Amount Handling

The puzzle does NOT enforce that `collateral_amount` equals the coin's actual amount:

```clojure
; Puzzle creates coin with solution-provided amount
(CREATE_COIN collateral_destination collateral_amount)

; The coin's actual amount may differ
; If collateral_amount < coin.amount: excess is destroyed (fee)
; If collateral_amount > coin.amount: spend fails (insufficient funds)
```

The driver SHOULD set `collateral_amount = coin.amount` to recover full collateral.

### Full Spend Bundle

```
Spend 1: Checkpoint Singleton (membership query path)
    Solution:
        - query_pubkey: VALIDATOR_PUBKEY
        - is_member: false
        - merkle_proof: [non-membership proof]
    Conditions:
        - CREATE_COIN: singleton recreation (unchanged)
        - CREATE_COIN_ANNOUNCEMENT: non-membership announcement

Spend 2: Registration Coin
    Solution:
        - epoch: current_epoch
        - collateral_destination: validator_wallet_puzzle_hash
        - collateral_amount: COLLATERAL_AMOUNT
        - conditions: []
    Conditions:
        - ASSERT_COIN_ANNOUNCEMENT: non-membership from checkpoint
        - CREATE_COIN: collateral to destination
```

### Transaction Atomicity

Both spends are in the same bundle:
- Either both succeed or both fail
- No partial execution
- Announcement and assertion happen atomically

### Puzzle Source

```clojure
(mod (
    VALIDATOR_PUBKEY
    CHECKPOINT_SINGLETON_ID

    epoch
    collateral_destination
    collateral_amount
    conditions
)
    ; ... announcement assertion ...

    (c
        (list ASSERT_COIN_ANNOUNCEMENT announcement_hash)
        (c
            (list CREATE_COIN collateral_destination collateral_amount)
            conditions  ; Additional conditions from solution
        )
    )
)
```

### Additional Conditions

The `conditions` parameter allows validators to add:
- Memos (for tracking/identification)
- Time locks (unusual but possible)
- Additional assertions

The puzzle appends these to the standard conditions.

## Acceptance Criteria

- [ ] CREATE_COIN condition uses solution-provided destination
- [ ] CREATE_COIN condition uses solution-provided amount
- [ ] Driver sets amount equal to coin's actual amount
- [ ] Destination can be any valid puzzle hash
- [ ] Spend succeeds with valid announcement and sufficient amount
- [ ] Spend fails if amount exceeds coin's holdings
- [ ] Additional conditions from solution are included

## Implementation Notes

- **Primary codebase:** `puzzles/registration_coin.clsp`
- **Driver code:** `src/registration_coin.rs` — `build_recovery_spend()`

### Driver Implementation

```rust
fn build_collateral_recovery(
    registration_coin: &Coin,
    destination: Bytes32,
    current_epoch: u64,
) -> CoinSpend {
    let solution = RegistrationCoinSolution {
        epoch: current_epoch,
        collateral_destination: destination,
        collateral_amount: registration_coin.amount,  // Full recovery
        conditions: vec![],
    };

    CoinSpend {
        coin: registration_coin.clone(),
        puzzle_reveal: compute_registration_puzzle(
            registration_coin.pubkey,
            checkpoint_singleton_id,
        ),
        solution: solution.to_clvm(),
    }
}
```

## Verification

1. Register validator with known collateral amount (e.g., 1 XCH)
2. Exit validator via checkpoint
3. Build recovery spend with destination = test wallet
4. Submit spend bundle
5. Query test wallet; verify received exactly 1 XCH
6. Verify registration coin is spent (no longer exists)
7. Test with partial amount recovery; verify difference goes to fees
