# REG-001 — Registration Coin Puzzle Structure

> **Authoritative requirement:** [REG-001](../NORMATIVE.md#REG-001)
> **Verification:** [VERIFICATION.md](../VERIFICATION.md)
> **Tracking:** [TRACKING.yaml](../TRACKING.yaml)
> **CHIP reference:** [chip-groth16-l2-consensus.md](../../../../resources/chip-groth16-l2-consensus.md) — Part 2: Registration Coin

## Summary

The registration coin puzzle commits to exactly two curried parameters that bind the coin to a specific validator and checkpoint singleton. This structure enables deterministic puzzle hash computation and ensures the coin can only be spent when the correct checkpoint singleton confirms the validator's exit.

## Specification

### Curried Parameters

| Parameter | Type | Size | Description |
|-----------|------|------|-------------|
| `VALIDATOR_PUBKEY` | PublicKey | 48 bytes | BLS12-381 G1 point, compressed, ZCash format |
| `CHECKPOINT_SINGLETON_ID` | Bytes32 | 32 bytes | **Coin ID** of the current checkpoint singleton (not launcher ID) |

### Parameter Details

**VALIDATOR_PUBKEY:**
- The BLS public key that identifies this validator
- Used to verify the validator is the subject of membership announcements
- Must match the pubkey used during registration via network coin

**CHECKPOINT_SINGLETON_ID:**
- The coin ID of the checkpoint singleton at the time of registration
- Used in `AssertCoinAnnouncement` to bind to the correct singleton
- Changes with each checkpoint spend (new singleton coin ID)

### Important: Coin ID vs Launcher ID

The `CHECKPOINT_SINGLETON_ID` is the **coin ID**, not the launcher ID:
- Launcher ID: Permanent identifier for the singleton lineage
- Coin ID: Identifier of the current singleton coin instance

This distinction matters because `AssertCoinAnnouncement` requires the announcing coin's ID, not its launcher ID. The registration coin must assert announcements from the specific checkpoint singleton coin.

### Puzzle Hash Computation

```python
def registration_coin_puzzle_hash(pubkey: bytes, checkpoint_id: bytes) -> bytes:
    """Compute the puzzle hash for a registration coin"""
    return curry_hash(
        REGISTRATION_COIN_MOD_HASH,
        pubkey,           # 48 bytes
        checkpoint_id     # 32 bytes
    )
```

### Puzzle Template (Chialisp)

```clojure
(mod (
    VALIDATOR_PUBKEY          ; Curried: 48-byte BLS pubkey
    CHECKPOINT_SINGLETON_ID   ; Curried: 32-byte coin ID

    ; Solution parameters
    epoch
    collateral_destination
    collateral_amount
    conditions
)
    ; Puzzle logic...
)
```

### Why Exactly Two Parameters

The minimal currying serves specific purposes:
1. `VALIDATOR_PUBKEY`: Links the coin to a specific validator identity
2. `CHECKPOINT_SINGLETON_ID`: Links the coin to the correct checkpoint authority

Additional parameters would:
- Increase puzzle complexity
- Make puzzle hash computation more expensive
- Potentially introduce security vulnerabilities

Parameters that might seem useful but are intentionally excluded:
- `COLLATERAL_AMOUNT`: Passed in solution, not curried (allows flexibility)
- `NETWORK_ID`: Implicit via checkpoint singleton

## Acceptance Criteria

- [ ] Puzzle accepts exactly 2 curried parameters
- [ ] First parameter is 48-byte pubkey
- [ ] Second parameter is 32-byte checkpoint singleton coin ID
- [ ] Puzzle hash is deterministically computable from parameters
- [ ] Decurrying returns exactly these two parameters
- [ ] Puzzle fails if parameters are missing or wrong size

## Implementation Notes

- **Puzzle source:** `puzzles/registration_coin.rue` (Rue)
- **Compiled hex:** `puzzles/compiled/registration_coin.hex` (CLVM bytecode)
- **Compiled hash:** `puzzles/compiled/registration_coin.hash` (tree hash)
- **Driver code:** `src/puzzles/registration_coin.rs`
- **Dependencies:** `curry_hash` utility, `REGISTRATION_COIN_MOD_HASH`

### Compile Pipeline

```
puzzles/registration_coin.rue
   │  rue build -x  → puzzles/compiled/registration_coin.hex
   │  rue build --hash → puzzles/compiled/registration_coin.hash
   ▼
src/puzzles/registration_coin.rs
   │  include_str!("../../puzzles/compiled/registration_coin.hex")
   │  include_str!("../../puzzles/compiled/registration_coin.hash")
   ▼
REGISTRATION_COIN_PUZZLE_HEX   — CLVM bytecode embedded at compile time
REGISTRATION_COIN_MOD_HASH_HEX — tree hash used by network coin and indexer
```

The `.hex` file contains the serialized CLVM bytecode of the uncurried puzzle.
The `.hash` file contains the tree hash (0x-prefixed 32-byte hex), which is the
`REGISTRATION_COIN_MOD_HASH` used by:
- The network coin to compute the curried puzzle hash for each validator (NET-003)
- The indexer for lineage verification (IDX-002)

### Rebuilding Compiled Artifacts

```bash
rue build -x puzzles/registration_coin.rue > puzzles/compiled/registration_coin.hex
rue build --hash puzzles/registration_coin.rue > puzzles/compiled/registration_coin.hash
```
