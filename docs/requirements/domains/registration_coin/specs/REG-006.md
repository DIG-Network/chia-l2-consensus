# REG-006 — Epoch Replay Protection

> **Authoritative requirement:** [REG-006](../NORMATIVE.md#REG-006)
> **Verification:** [VERIFICATION.md](../VERIFICATION.md)
> **Tracking:** [TRACKING.yaml](../TRACKING.yaml)
> **CHIP reference:** [chip-groth16-l2-consensus.md](../../../../resources/chip-groth16-l2-consensus.md) — Part 2: Registration Coin

## Summary

The registration coin includes the checkpoint epoch in its announcement assertion, preventing replay attacks where an old non-membership announcement is used to recover collateral after the validator has rejoined the network.

## Specification

### Attack Scenario Without Epoch Protection

```
1. Validator A is active in epoch 5
2. Validator A exits; epoch 6 checkpoint excludes them
3. Validator A recovers collateral using epoch 6 non-membership announcement
4. Wait... Validator A decides not to recover yet
5. Validator A re-registers, becomes active in epoch 7
6. Validator A tries to use the old epoch 6 announcement to recover collateral
7. WITHOUT epoch check: Success! Validator steals their own collateral while still active
```

### How Epoch Protection Works

The registration coin's announcement assertion includes the epoch:

```
inner_announcement = sha256(
    "membership"
    + epoch_be8           # THIS epoch, not any epoch
    + pubkey
    + is_member
)
```

The checkpoint singleton only announces with its current epoch:

```
checkpoint_epoch = 8  # Current epoch
announcement = sha256("membership" + 8 + pubkey + 0x00)
```

If the registration coin expects epoch 6 but checkpoint is at epoch 8:
- Registration coin asserts: sha256("membership" + 6 + ...)
- Checkpoint announces: sha256("membership" + 8 + ...)
- Hashes don't match → spend fails

### Epoch Lifecycle

```
Epoch 5: Validator A active
         ├── A in Merkle tree
         └── A cannot recover (membership=true)

Epoch 6: Validator A exits
         ├── A removed from Merkle tree
         ├── A can query non-membership
         └── A can recover with epoch=6 announcement

Epoch 7: Validator A re-registers
         ├── A back in Merkle tree
         ├── A cannot query non-membership (proof fails)
         └── Old epoch=6 announcement won't work (epoch mismatch)

Epoch 8: Current
         ├── A still active
         └── A cannot recover until exits again
```

### Why the Solution Provides Epoch

The registration coin puzzle doesn't store the current epoch — it's provided in the solution:

```clojure
(mod (
    VALIDATOR_PUBKEY
    CHECKPOINT_SINGLETON_ID

    epoch                    ; From solution, not curried
    collateral_destination
    collateral_amount
    conditions
)
    ; Use epoch in announcement computation
    (sha256 "membership" (int_to_bytes epoch 8) VALIDATOR_PUBKEY 0x00)
)
```

The validator must provide the correct epoch (matching checkpoint's current epoch) for the assertion to succeed. An attacker cannot:
- Provide old epoch → checkpoint won't emit matching announcement
- Provide future epoch → checkpoint won't emit matching announcement
- Provide current epoch while active → checkpoint emits membership=true, not membership=false

### Checkpoint Singleton Epoch Tracking

The checkpoint singleton maintains epoch as state:

```clojure
; Checkpoint singleton state (curried in)
state_root
epoch                    ; Increments by 1 each checkpoint
validator_merkle_root
validator_count
```

Membership query uses current epoch:
```clojure
; Membership query spend path
(CREATE_COIN_ANNOUNCEMENT
    (sha256 "membership"
            (int_to_bytes epoch 8)      ; Current epoch
            query_pubkey
            is_member))
```

### Edge Case: Rapid Re-registration

```
Block N:   Checkpoint 6 accepted (A excluded)
Block N+1: A queries non-membership at epoch 6
Block N+2: A re-registers via network coin
Block N+3: Checkpoint 7 accepted (A included)
```

If A queries non-membership in block N+1 but hasn't spent registration coin yet:
- The announcement is valid for epoch 6
- But A hasn't spent yet, so collateral is still locked
- Once A is included in epoch 7, their registration coin at epoch 6 state no longer works
- A would need the NEW checkpoint singleton coin ID anyway (it changed at epoch 7)

This edge case is handled by the fact that `CHECKPOINT_SINGLETON_ID` is a coin ID, not launcher ID.

## Acceptance Criteria

- [ ] Announcement includes epoch as 8-byte big-endian u64
- [ ] Solution epoch must match checkpoint's current epoch
- [ ] Spend with old epoch fails (hash mismatch)
- [ ] Spend with future epoch fails (hash mismatch)
- [ ] Re-registered validator cannot use old announcement
- [ ] Epoch increments exactly by 1 per checkpoint

## Implementation Notes

- **Registration coin:** Computes announcement with solution-provided epoch
- **Checkpoint singleton:** Emits announcement with its curried epoch
- **Driver:** Must query current epoch before building spend

### Common Mistake

Using a cached/stale epoch value:

```rust
// WRONG: Using old epoch
let announcement = compute_announcement(validator_pk, cached_epoch);

// RIGHT: Query current epoch
let current_epoch = client.epoch();  // Fresh from checkpoint
let announcement = compute_announcement(validator_pk, current_epoch);
```

## Verification

1. Register validator at epoch N
2. Exit at epoch N+1
3. DO NOT recover collateral
4. Re-register, become active at epoch N+2
5. Attempt recovery with epoch N+1 value → verify failure
6. Exit again at epoch N+3
7. Recover with epoch N+3 → verify success
