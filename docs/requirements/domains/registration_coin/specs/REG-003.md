# REG-003 — Collateral Lock

> **Authoritative requirement:** [REG-003](../NORMATIVE.md#REG-003)
> **Verification:** [VERIFICATION.md](../VERIFICATION.md)
> **Tracking:** [TRACKING.yaml](../TRACKING.yaml)

## Summary

The registration coin holds the validator's collateral and MUST remain locked until the validator has exited the active set. The only way to unlock the collateral is by proving non-membership via a checkpoint singleton announcement. This creates economic incentives for validators to behave honestly and participate in the exit process.

## Specification

### Collateral Properties

| Property | Value | Source |
|----------|-------|--------|
| Amount | `COLLATERAL_AMOUNT` | Curried into network coin, fixed at deployment |
| Denomination | Mojos | Chia standard (1 XCH = 10^12 mojos) |
| Lock condition | Non-membership announcement | Checkpoint singleton spend path 2 |
| Unlock trigger | Checkpoint excludes validator | L2 consensus decision |

### Lock Mechanism

The registration coin puzzle enforces the lock by requiring:

```clojure
(ASSERT_COIN_ANNOUNCEMENT
    (sha256 CHECKPOINT_SINGLETON_ID
        (sha256 "membership" epoch_be8 VALIDATOR_PUBKEY 0x00)))
```

This condition can only be satisfied if:
1. The checkpoint singleton emits the exact announcement
2. The announcement indicates non-membership (`0x00`)
3. The epoch matches (replay protection)

### Why Collateral Matters

Collateral serves multiple purposes:

1. **Skin in the game**: Validators have economic stake in honest behavior
2. **Slashing deterrent**: Misbehavior can result in collateral loss
3. **Exit coordination**: Validators must coordinate with the network to exit
4. **Sybil resistance**: Registration has real economic cost

### Lock Duration

The collateral remains locked from registration until:
1. Validator signals exit intent at L2 level
2. Next checkpoint excludes validator from Merkle tree
3. Validator queries membership (proves non-membership)
4. Registration coin is spent, returning collateral

There is no time-based unlock — only consensus-based unlock.

### Spending Without Announcement (Impossible)

Attempting to spend the registration coin without the announcement:

```
Spend bundle:
    - Registration coin spend (no announcement)

Result: ASSERT_COIN_ANNOUNCEMENT fails
        Spend bundle rejected by mempool
```

### Spending With Membership Announcement (Impossible)

Attempting to spend while still a member:

```
Spend bundle:
    - Checkpoint singleton membership query (is_member=true)
    - Registration coin spend

Result: Registration coin asserts is_member=0x00
        Checkpoint announces is_member=0x01
        Announcement mismatch → spend fails
```

### Valid Unlock Flow

```
1. Validator signals exit to L2
2. Next checkpoint excludes validator (Merkle root changes)
3. Validator calls checkpoint membership query:
   - Provides non-membership Merkle proof
   - Checkpoint emits: sha256("membership" + epoch + pubkey + 0x00)
4. Same spend bundle includes registration coin spend:
   - Asserts the non-membership announcement
   - Creates coin at collateral_destination
5. Collateral returned to validator
```

## Acceptance Criteria

- [ ] Registration coin holds exactly `COLLATERAL_AMOUNT` mojos
- [ ] Spend without checkpoint announcement fails
- [ ] Spend with membership=true announcement fails
- [ ] Spend with membership=false announcement succeeds
- [ ] Collateral cannot be partially withdrawn
- [ ] Collateral amount matches network configuration

## Implementation Notes

- **Primary codebase:** `puzzles/registration_coin.clsp`
- **Collateral amount:** Set in `NetworkConfig`, curried into network coin

### Collateral Amount Considerations

Typical values:
- Testnet: 0.001 XCH (1,000,000,000 mojos)
- Mainnet: 1-10 XCH (depends on L2 requirements)

The amount should be:
- High enough to deter frivolous registration
- Low enough to not exclude legitimate validators
- Proportional to potential slashing penalties

## Verification

1. Register validator with known collateral amount
2. Query registration coin; verify amount = `COLLATERAL_AMOUNT`
3. Attempt spend without any announcement → verify failure
4. Attempt spend with is_member=true announcement → verify failure
5. Generate valid non-membership announcement
6. Spend registration coin with valid announcement → verify success
7. Verify destination receives full collateral amount
