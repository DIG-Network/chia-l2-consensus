# REG-002 — Lineage Verification

> **Authoritative requirement:** [REG-002](../NORMATIVE.md#REG-002)
> **Verification:** [VERIFICATION.md](../VERIFICATION.md)
> **Tracking:** [TRACKING.yaml](../TRACKING.yaml)
> **CHIP reference:** [chip-groth16-l2-consensus.md](../../../../resources/chip-groth16-l2-consensus.md) — Part 2: Registration Coin

## Summary

A registration coin is only valid if it was created by a legitimate network coin spend. This lineage verification prevents attackers from creating fake registration coins directly. The verification is performed off-chain by the indexer because on-chain lineage proofs would be prohibitively expensive.

## Specification

### Lineage Definition

A valid registration coin has:
```
parent_coin_id = sha256(
    network_coin_parent_id,
    network_coin_puzzle_hash,
    network_coin_amount          # 1 mojo
)
```

Where the parent must be a spent network coin.

### Verification Process

The indexer maintains a set of valid network coin spends:

```python
class Indexer:
    network_coin_spend_ids: Set[Bytes32]  # All spent network coin IDs

    def verify_lineage(self, coin: Coin) -> bool:
        """Verify registration coin was created by network coin"""
        return coin.parent_coin_id in self.network_coin_spend_ids
```

### What Makes a Network Coin Spend Valid

A coin spend is a valid network coin spend if:
1. The spent coin has the network coin puzzle hash
2. The spent coin's parent traces back to the network coin launcher
3. The spent coin's amount is 1 mojo (singleton convention)

### Indexer Detection Algorithm

```python
def process_new_coin(self, coin: Coin, parent_spend: CoinSpend):
    # Check if parent was a network coin
    if parent_spend.coin.puzzle_hash != NETWORK_COIN_PUZZLE_HASH:
        return  # Not created by network coin

    # Verify lineage of parent
    if not self.is_valid_network_coin(parent_spend.coin):
        return  # Parent isn't a valid network coin

    # Extract pubkey from memo
    pubkey = self.extract_pubkey_memo(parent_spend)
    if pubkey is None or len(pubkey) != 48:
        return  # No valid pubkey memo

    # Verify puzzle hash matches
    expected_hash = curry_hash(
        REGISTRATION_COIN_MOD_HASH,
        pubkey,
        CHECKPOINT_SINGLETON_ID
    )
    if coin.puzzle_hash != expected_hash:
        return  # Puzzle hash mismatch

    # Valid registration coin
    self.register_validator(pubkey, coin)
```

### Why Off-Chain Verification

On-chain lineage proofs would require:
1. Passing the full parent coin record in the solution
2. Verifying parent puzzle hash matches network coin
3. Recursively verifying parent's parent (singleton lineage)

This would cost millions of CLVM units per verification and bloat transaction sizes.

Off-chain verification is secure because:
1. Coin IDs are deterministic (sha256 of parent + puzzle_hash + amount)
2. The blockchain is immutable — parent coins cannot be retroactively changed
3. The indexer can independently verify by querying the chain

### Attack Scenario Prevented

Without lineage verification:
1. Attacker creates coin with registration coin puzzle hash (curried with their pubkey)
2. Coin appears to be a valid registration coin
3. Attacker is counted as a validator without going through network coin

With lineage verification:
1. Attacker creates coin with registration coin puzzle hash
2. Indexer checks parent: not a network coin spend
3. Coin is ignored — attacker is not counted as validator

## Acceptance Criteria

- [ ] Indexer maintains set of all network coin spend IDs
- [ ] Coins with registration puzzle hash are verified for lineage
- [ ] Coins created by network coin spend are added to validator set
- [ ] Coins not created by network coin are ignored
- [ ] Lineage check includes singleton verification of network coin
- [ ] False registration coins produce no security impact

## Implementation Notes

- **Primary codebase:** `src/indexer.rs`
- **Not enforced on-chain:** Pure indexer logic
- **Performance:** O(1) lookup in spend ID set

### Data Structures

```rust
struct Indexer {
    /// All valid network coin spend coin IDs
    network_coin_spends: HashSet<Bytes32>,

    /// Currently active registration coins by pubkey
    registration_coins: HashMap<PublicKey, Coin>,

    /// Network coin puzzle hash (constant)
    network_coin_puzzle_hash: Bytes32,
}
```

## Verification

1. Deploy network coin and register validator normally
2. Verify indexer includes validator in set
3. Create coin with registration puzzle hash directly (no network coin)
4. Verify indexer does NOT include this coin
5. Query validator set; verify only legitimately registered validators appear
6. Attempt to use fake registration coin for checkpoint; verify failure
