# VAL-002 — Registration

> **Authoritative requirement:** [VAL-002](../NORMATIVE.md#VAL-002)
> **Verification:** [VERIFICATION.md](../VERIFICATION.md)
> **Tracking:** [TRACKING.yaml](../TRACKING.yaml)
> **CHIP reference:** [chip-groth16-l2-consensus.md](../../../../resources/chip-groth16-l2-consensus.md) — Part 1: Network Coin

## Summary

To register, a validator spends the network coin with their secret key, providing collateral at least equal to COLLATERAL_AMOUNT, and waits for indexer confirmation before participating in consensus.

## Specification

### Registration Process

1. Generate BLS keypair (VAL-001)
2. Sync with network to get current state
3. Spend network coin with:
   - Signature proving key ownership
   - Collateral locked in registration coin
   - Pubkey included as memo for indexing
4. Wait for confirmation
5. Verify registration indexed
6. Wait for next checkpoint to become active

### Registration Message

```
registration_message = sha256("register" + pubkey)
agg_sig_me_message = registration_message + genesis_challenge + network_coin_coin_id
```

The validator signs this message to prove they control the private key.

### Collateral Requirements

- Amount must equal or exceed COLLATERAL_AMOUNT from network config
- Locked in registration coin until exit confirmed
- Cannot be partially withdrawn

### Post-Registration States

| State | Meaning |
|-------|---------|
| PENDING | Registration indexed but not yet in checkpoint |
| ACTIVE | Included in current validator_merkle_root |

## Acceptance Criteria

- [x] Validator can build registration signing message (`vv_req_val_002_signing_message_is_96_bytes`)
- [x] Signature proves key ownership (`vv_req_val_002_signature_verifies`)
- [x] Wrong key/challenge/coin_id rejected (`vv_req_val_002_wrong_key_fails`, `vv_req_val_002_wrong_genesis_challenge_fails`, `vv_req_val_002_wrong_coin_id_fails`)
- [x] Message format matches wire spec (`vv_req_val_002_registration_message_matches_wire_spec`)
- [x] Collateral amount accessible from config (`vv_req_val_002_collateral_from_config`)
- [x] Signing is deterministic (`vv_req_val_002_signing_deterministic`)

## Implementation Notes

- **Tool:** `l2-validator register`
- **API:** `ConsensusClient.register_validator()`
- **Cost:** ~5.3M CLVM units

### Registration Command

```bash
./target/release/l2-validator register \
  --config validator_config.toml \
  --collateral-coin <coin_id_of_your_xch_coin>

# Output:
# Building registration spend bundle...
# Submit this bundle? [y/N]: y
# Submitting to node...
# Bundle ID: <hex>
# Registration submitted. Wait for confirmation.
```

### Verification

```bash
./target/release/l2-validator status --config validator_config.toml

# Expected output:
# Your public key: <hex>
# Registration status: REGISTERED
# Registration coin: <coin_id>
# Active in validator set: PENDING
```

## Verification

1. Generate keypair
2. Fund wallet with collateral + fees
3. Submit registration
4. Verify bundle accepted
5. Verify indexer detects registration
6. Verify status shows PENDING then ACTIVE
