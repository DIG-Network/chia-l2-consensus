# VAL-005 — Forced Exit

> **Authoritative requirement:** [VAL-005](../NORMATIVE.md#VAL-005)
> **Verification:** [VERIFICATION.md](../VERIFICATION.md)
> **Tracking:** [TRACKING.yaml](../TRACKING.yaml)
> **CHIP reference:** [chip-groth16-l2-consensus.md](../../../../resources/chip-groth16-l2-consensus.md) — De-Registration and Collateral Recovery

## Summary

A validator may be force-exited by majority vote. The next checkpoint excludes them and the same collateral recovery process applies, potentially with slashing to a governance address.

## Specification

### Forced Exit Scenarios

| Scenario | Reason |
|----------|--------|
| Key compromise | Attacker could use key for malicious voting |
| Validator offline | Cannot contribute to majority |
| Misbehavior | Signed conflicting checkpoints |
| Governance decision | Network policy enforcement |

### Forced Exit Process

1. **Majority decision**: L2 governance decides to exclude validator
2. **Checkpoint preparation**: Next checkpoint omits validator from Merkle root
3. **Checkpoint submission**: Majority signs and submits checkpoint
4. **Exclusion confirmed**: Validator's pubkey not in new Merkle root

### Collateral Handling

After forced exit, the registration coin can be spent by anyone with:
- Valid non-membership proof
- Correct announcement assertion

The `collateral_destination` in the solution determines where collateral goes:
- Normal exit: Validator's address
- Slashing: Governance/burn address

### Slashing Extension

The registration coin's `conditions` field enables slashing:
- L2 level defines slashing conditions
- Forced exit can redirect collateral
- Not enforced on L1 (requires L2 coordination)

## Acceptance Criteria

- [ ] Majority can exclude any validator
- [ ] Excluded validator cannot sign after exclusion
- [ ] Collateral recovery uses same process as voluntary
- [ ] Slashing destination can be specified
- [ ] Forced exit is documented in governance

## Implementation Notes

- **Governance:** L2 level decision
- **Execution:** Standard checkpoint with modified Merkle root
- **Recovery:** Same as VAL-004, different destination possible

### Forced Exit vs Voluntary

| Aspect | Voluntary | Forced |
|--------|-----------|--------|
| Initiation | Validator signals | Majority decision |
| Cooperation | Validator participates | May be hostile |
| Destination | Validator chooses | Governance may override |
| Timing | Validator controls | Majority controls |

### Slashing Implementation

```rust
// In registration coin spend solution
let solution = RegistrationCoinSolution {
    epoch: current_epoch,
    collateral_destination: governance_address,  // Slashing destination
    collateral_amount: config.collateral_amount,
    conditions: vec![],  // Additional L2 conditions
};
```

## Verification

1. Register test validator
2. Simulate governance forced exit decision
3. Submit checkpoint excluding validator
4. Verify validator status shows INACTIVE
5. Test collateral recovery to governance address
6. Verify original validator cannot recover
