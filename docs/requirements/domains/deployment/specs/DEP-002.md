# DEP-002 — Genesis Coin

> **Authoritative requirement:** [DEP-002](../NORMATIVE.md#DEP-002)
> **Verification:** [VERIFICATION.md](../VERIFICATION.md)
> **Tracking:** [TRACKING.yaml](../TRACKING.yaml)

## Summary

Deployment uses a genesis coin (minimum 2 XCH recommended) from which both singleton launcher IDs are derived. This resolves circular dependencies between the network coin and checkpoint singleton.

## Specification

### Why Genesis Coin

Both singletons need to know each other's launcher IDs:
- Network coin needs checkpoint launcher ID for registration coins
- Checkpoint singleton needs its own ID for state tracking

Without a shared genesis, there would be a circular dependency. The genesis coin resolves this by deriving both launcher IDs from a single source.

### Deployment Flow

```
Genesis Coin
    │
    ├─► Network Coin Launcher ─► Network Coin
    │
    └─► Checkpoint Launcher ─► Checkpoint Singleton
```

### Genesis Coin Requirements

| Requirement | Reason |
|-------------|--------|
| Minimum 2 XCH | Cover two launcher spends + fees |
| Single owner | Coordinated deployment |
| Unspent | Cannot be already used |

### Launcher ID Derivation

```rust
// Both IDs derived from genesis coin spend
let network_coin_launcher_id = sha256(genesis_coin_id + network_coin_launcher_puzzle_hash);
let checkpoint_launcher_id = sha256(genesis_coin_id + checkpoint_launcher_puzzle_hash);
```

## Acceptance Criteria

- [x] Genesis coin has sufficient value (>= 1 mojo per singleton) (`vv_req_dep_002_deploy_both_singletons`)
- [x] Both launcher IDs derived from genesis (`vv_req_dep_002_ids_predictable_before_deploy`)
- [x] Network coin knows checkpoint launcher ID — via NetworkConfig (`vv_req_dep_002_config_fields_populated`)
- [x] Checkpoint singleton knows its launcher ID — via NetworkConfig (`vv_req_dep_002_config_fields_populated`)
- [x] Deployment is atomic (both singletons created together) (`vv_req_dep_002_both_created_in_same_block`)

## Implementation Notes

- **Tool:** `l2-consensus-deploy`
- **Input:** Genesis coin ID, VK, configuration
- **Output:** Both singletons deployed

### Deployment Script

```bash
./target/release/l2-consensus-deploy \
  --genesis-coin <coin_id> \
  --vk-path verification_key.bin \
  --config network_config.json \
  --tree-depth 32 \
  --collateral-amount 10000000000000
```

### Spend Bundle Structure

The deployment creates a spend bundle with:
1. Genesis coin spend (creates two child coins)
2. Network coin launcher spend
3. Checkpoint singleton launcher spend

All three spends are atomic - all succeed or all fail.

## Verification

1. Verify genesis coin exists and is unspent
2. Verify genesis coin amount >= 2 XCH
3. Deploy and verify both singletons created
4. Verify launcher IDs match derivation
5. Verify cross-references are correct
