# SEC-008 — Condition Injection Protection

> **Authoritative requirement:** [SEC-008](../NORMATIVE.md#SEC-008)
> **Verification:** [VERIFICATION.md](../VERIFICATION.md)
> **Tracking:** [TRACKING.yaml](../TRACKING.yaml)
> **Source:** Chia Knowledge Graph — condition-injection.md, solution-manipulation.md

## Summary

All three Rue puzzles currently pass through `...conditions` from the
solution without validation. A malicious farmer could inject extra
CREATE_COIN, RESERVE_FEE, or AGG_SIG_UNSAFE conditions to steal funds
or manipulate the protocol.

## Specification

### Current Risk

```rue
// In all three puzzles:
[agg_sig, create_registration, recreate_self, ...conditions]
```

The `...conditions` spread appends ANY conditions from the solution.
This includes potentially dangerous opcodes:
- `CREATE_COIN` → create coins to attacker's address
- `RESERVE_FEE` → drain funds to miner fees
- `AGG_SIG_UNSAFE` → create replayable signatures

### Fix Options

**Option A: Remove passthrough entirely**
- Remove `conditions` parameter from all puzzles
- Fees handled at the outer spend bundle level (P2 coin pays fees)
- Simplest and most secure

**Option B: Whitelist safe opcodes**
- Only allow: ASSERT_* conditions (opcodes 60-87)
- Block: CREATE_COIN (51), AGG_SIG_* (49-50), RESERVE_FEE (52)
- More flexible but complex

**Option C: Sign conditions**
- `AggSigMe pk sha256tree(conditions)`
- Conditions are signed so farmer can't modify them
- Requires signer to approve every condition

### Recommended: Option A for registration_coin and checkpoint_inner

The registration coin and checkpoint inner puzzles don't need passthrough
conditions — all required conditions are generated by the puzzle logic.
Remove `conditions` parameter entirely.

For network_coin_inner, keep `conditions` but document that the singleton
owner controls it via `AggSigMe` (the validator's signature already covers
the entire spend).

## Acceptance Criteria

- [ ] Registration coin: remove `...conditions` passthrough OR sign them
- [ ] Checkpoint inner: remove `...conditions` passthrough OR sign them
- [ ] Network coin: document that AggSigMe protects conditions
- [ ] No AGG_SIG_UNSAFE injectable via conditions in any puzzle
- [ ] Test: inject malicious condition → verify rejection or no effect
