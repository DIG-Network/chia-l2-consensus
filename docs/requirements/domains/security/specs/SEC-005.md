# SEC-005 — Lineage Enforcement

## Summary

Registration coin lineage verification is enforced by the indexer. Coins without valid lineage are excluded from the validator set regardless of puzzle hash. This prevents fake registrations.

## Specification

### Lineage Requirement

A registration coin is only valid if its parent coin ID traces back to a network coin spend. This is verified off-chain by the indexer.

### Why Lineage Matters

Without lineage verification:
- Anyone could create a coin with the registration puzzle hash
- Fake validators could join without going through proper registration
- The network coin gatekeeper would be bypassed
- Economic security (collateral) could be undermined

### Enforcement Point

| Layer | Enforcement |
|-------|-------------|
| On-chain | Not enforced (would require expensive iteration) |
| Indexer | Strictly enforced |
| Validator signing | Relies on correct indexer state |

### Why This Is Secure

- Coin IDs are deterministic (hash of parent + puzzle + amount)
- Blockchain is immutable
- Correct lineage cannot be faked
- Majority validators independently verify lineage

## Acceptance Criteria

- [ ] Indexer checks parent is network coin spend
- [ ] Coins with wrong parent are ignored
- [ ] Puzzle hash alone is insufficient for inclusion
- [ ] Majority validators independently verify
- [ ] Documentation explains lineage security

## Implementation Notes

- **Enforcer:** `src/indexer/validator_set.rs`
- **Check:** `try_parse_registration_coin()`

### Lineage Check

```rust
// Step 1: Check if parent is a known network coin spend
let is_valid_parent = self.network_coin_spend_ids.contains(
    &coin.parent_coin_info
);
if !is_valid_parent {
    return Ok(None);  // Silently ignore invalid coin
}
```

### Security Properties

1. **Determinism**: Coin ID = sha256(parent_id + puzzle_hash + amount)
2. **Immutability**: Once confirmed, parent cannot change
3. **Verifiability**: Any node can verify lineage independently
4. **Consensus**: Majority must agree on validator set

## Verification

1. Register through network coin -> included in validator set
2. Create coin with same puzzle hash, wrong parent -> excluded
3. Verify all indexed registration coins have valid lineage
4. Verify Merkle root only includes valid registrations
