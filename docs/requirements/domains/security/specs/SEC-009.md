# SEC-009 — Registration Coin Destination Binding

> **Authoritative requirement:** [SEC-009](../NORMATIVE.md#SEC-009)
> **Verification:** [VERIFICATION.md](../VERIFICATION.md)
> **Tracking:** [TRACKING.yaml](../TRACKING.yaml)
> **Source:** Chia Knowledge Graph — rbf-attack-unsecured-destination.md, solution-manipulation.md

## Summary

The registration coin's `collateral_destination` and `collateral_amount` are
provided in the solution and not signed or bound. A malicious farmer could
redirect collateral recovery to their own address via RBF (Replace-By-Fee)
attack in the mempool.

## Specification

### Current Risk

```rue
// registration_coin.rue lines 57-62
let create_collateral = CreateCoin {
    puzzle_hash: collateral_destination,  // FROM SOLUTION — attacker-controllable
    amount: collateral_amount,            // FROM SOLUTION — attacker-controllable
    memos: nil,
};
```

### Attack Scenario

1. Validator submits collateral recovery tx with `dest = validator_address`
2. Farmer sees tx in mempool
3. Farmer creates same spend with `dest = attacker_address`
4. Farmer adds higher fee → miner picks farmer's version
5. Collateral goes to attacker

### Fix Options

**Option A: Require AggSigMe on destination**
- Add validator pubkey signature over `sha256(collateral_destination + collateral_amount)`
- Farmer can't produce valid signature for different destination
- Requires registration coin to have AGG_SIG_ME condition

**Option B: Curry destination at registration time**
- Validator specifies recovery address when creating registration coin
- Curried into puzzle hash → immutable
- Less flexible but no RBF risk

**Option C: Accept design — document as low risk**
- The registration coin requires a non-membership announcement
- The announcement must come from the correct checkpoint singleton
- An attacker needs the announcement AND the registration coin in the same bundle
- If the validator constructs the bundle privately and submits it,
  farmer attack requires seeing it in mempool first
- Document this as acceptable with mempool privacy recommendations

### Recommended: Option A or C

Option A adds AGG_SIG_ME with validator pubkey over the destination hash.
This is the most secure but adds complexity.

Option C documents the risk as mitigated by the announcement requirement
(farmer needs both the announcement and the registration coin spend in
one bundle).

## Acceptance Criteria

- [ ] Risk documented in puzzle comments
- [ ] Either: destination signed via AggSigMe, OR
- [ ] Risk analysis shows mempool attack is impractical due to announcement requirement
- [ ] Test: verify destination binding or document risk acceptance
