# SEC-007 — CLVM Vulnerability Audit

> **Authoritative requirement:** [SEC-007](../NORMATIVE.md#SEC-007)
> **Verification:** [VERIFICATION.md](../VERIFICATION.md)
> **Tracking:** [TRACKING.yaml](../TRACKING.yaml)
> **Source:** Chia Knowledge Graph vulnerability nuggets analysis

## Summary

This requirement verifies that all known Chialisp/CLVM vulnerabilities
documented in the Chia knowledge graph are addressed in the three Rue
puzzles (network_coin_inner.rue, registration_coin.rue, checkpoint_inner.rue).

## Vulnerabilities Assessed

### V1: CATbleed (Byte-Shifting in SHA-256 Concat) — MITIGATED

**Risk**: Manual `sha256(parent + puzzle_hash + amount)` allows byte shifting.
**Current state**: Puzzles use `curry_tree_hash()` and `sha256()` with typed
fields (Bytes32, PublicKey, Int). The `int_to_8_bytes_be()` helper ensures
fixed-width integer encoding. No manual coin ID computation — coin IDs are
managed by the singleton wrapper.
**Status**: MITIGATED by typed Rue fields and fixed-width encoding.

### V2: AGG_SIG_UNSAFE Replay — MITIGATED

**Risk**: `AGG_SIG_UNSAFE` signatures live on-chain forever, replayable.
**Current state**:
- `network_coin_inner.rue`: Uses `AggSigMe` (line 48) — bound to coin_id+genesis
- `registration_coin.rue`: No AGG_SIG at all (permissionless with proof)
- `checkpoint_inner.rue`: Uses `bls_verify` (line 342) — separate from AGG_SIG
**Status**: MITIGATED — no AGG_SIG_UNSAFE anywhere.

### V3: Announcement Replay — PARTIALLY MITIGATED

**Risk**: Announcement messages without unique identifiers can be replayed.
**Current state**:
- Registration coin asserts `sha256(CHECKPOINT_SINGLETON_ID + inner_announcement)` —
  bound to specific checkpoint coin ID
- Checkpoint emits announcement with epoch — bound to specific epoch
- BUT: no network_coin_launcher_id in announcements → see CHK-012
**Status**: PARTIALLY MITIGATED — epoch and coin_id binding exist; network ID missing.

### V4: Solution Manipulation / Farmer Attacks — GAP

**Risk**: Unsigned solution values can be modified by malicious farmers.
**Current state**:
- `network_coin_inner.rue`: `conditions` list from solution is passed through
  UNSIGNED. A farmer could inject extra conditions (CREATE_COIN to attacker).
- `registration_coin.rue`: `collateral_destination` and `collateral_amount`
  are from solution, UNSIGNED. A farmer could redirect collateral.
- `checkpoint_inner.rue`: `conditions` list is passed through UNSIGNED.
  `new_state_root` is from solution but bound via checkpoint_message+BLS signature.
**Status**: GAP — passthrough conditions and unsigned destinations need protection.

### V5: Unvalidated Passthrough Conditions — GAP

**Risk**: `...conditions` appended to output allows injection of dangerous
conditions (AGG_SIG_UNSAFE, extra CREATE_COIN, RESERVE_FEE).
**Current state**: All three puzzles append `...conditions` from solution without
filtering. An attacker in the mempool could inject:
- Extra CREATE_COIN conditions (stealing value)
- RESERVE_FEE (draining funds to fees)
**Status**: GAP — conditions should be signed or filtered.

### V6: Registration Coin Destination Not Signed — GAP

**Risk**: `collateral_destination` in registration_coin.rue is from solution
and not signed. A farmer could substitute their own address.
**Current state**: The registration coin has no AGG_SIG_ME — it's permissionless
(only needs the announcement). But the destination is attacker-controllable.
The TODO on line 29 of registration_coin.rue acknowledges this gap.
**Status**: GAP — destination should be bound (signed or curried).

### V7: Flash Loan Protection — NOT APPLICABLE

**Risk**: Ephemeral coins manipulating protocol state.
**Current state**: The checkpoint requires Groth16 proof + BLS majority signature.
Creating ephemeral validator registrations requires spending the network coin
(singleton, sequential). Flash loan attacks are not practical against this model.
**Status**: NOT APPLICABLE.

### V8: Bricked Coins — MITIGATED

**Risk**: Conditions that make coins permanently unspendable.
**Current state**:
- Registration coin: asserts announcement from a specific CHECKPOINT_SINGLETON_ID.
  If that ID changes (checkpoint advances), old registration coins may need
  updating. But the ID is curried at creation time and the checkpoint coin ID
  changes every epoch.
- NOTE: Registration coin is curried with CHECKPOINT_SINGLETON_ID which is
  a COIN ID, not launcher ID. This means it references a specific checkpoint
  instance. After that instance is spent, the coin can never satisfy its
  assertion unless the same coin ID reappears (impossible). This is a KNOWN
  DESIGN — the registration coin is only valid for the current checkpoint
  epoch. See REG-001 spec for details.
**Status**: BY DESIGN — registration coin lifetime tied to checkpoint epoch.

### V9: Network ID Not in Checkpoint Message — GAP (CHK-012)

**Risk**: Cross-network proof replay if two networks share same circuit.
**Current state**: checkpoint_message does not include network_coin_launcher_id.
**Status**: GAP — requirement CHK-012 created for this.

## Acceptance Criteria

- [ ] No AGG_SIG_UNSAFE in any puzzle (V2)
- [ ] All integer encoding uses fixed-width int_to_8_bytes_be (V1)
- [ ] Passthrough conditions reviewed for injection risk (V5)
- [ ] Registration coin destination binding documented (V6)
- [ ] Network ID binding planned (V9 → CHK-012)
- [ ] Announcement format includes binding fields (V3)
