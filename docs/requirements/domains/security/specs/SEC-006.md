# SEC-006 — Epoch Replay Protection

> **Authoritative requirement:** [SEC-006](../NORMATIVE.md#SEC-006)
> **Verification:** [VERIFICATION.md](../VERIFICATION.md)
> **Tracking:** [TRACKING.yaml](../TRACKING.yaml)
> **CHIP reference:** [chip-groth16-l2-consensus.md](../../../../resources/chip-groth16-l2-consensus.md) — Security

## Summary

The epoch included in membership announcements prevents replay attacks where old non-membership proofs are reused after a validator re-registers.

## Specification

### The Attack Scenario

Without epoch protection:
1. Validator A exits at epoch 5, receives non-membership proof
2. Validator A re-registers at epoch 6
3. Attacker replays old non-membership proof from epoch 5
4. Validator A's new registration is invalidly exited

### How Epoch Prevents This

The membership announcement includes the epoch number:

```
announcement_message = sha256(
    "membership"
    + epoch_be              (8 bytes)
    + pubkey
    + is_member
)
```

The registration coin must match the epoch in the announcement to the current checkpoint epoch.

### Epoch Matching

- Registration coin curries the checkpoint singleton coin ID at registration time
- Checkpoint singleton coin ID changes every checkpoint (new coin created)
- Announcement uses current checkpoint singleton coin ID
- Old announcements have wrong coin ID and fail assertion

### Additional Protection

Even if announcement format matched, the coin ID changes:
```
AssertCoinAnnouncement(sha256(checkpoint_singleton_coin_id + announcement_message))
```

The `checkpoint_singleton_coin_id` changes on every checkpoint spend.

## Acceptance Criteria

- [x] Membership announcement includes epoch as 8-byte BE (`vv_req_sec_006_epoch_is_8_byte_be`)
- [x] Different epochs produce different announcements (`vv_req_sec_006_different_epochs_different_inner`, `vv_req_sec_006_adjacent_epochs`)
- [x] Coin ID change provides additional protection (`vv_req_sec_006_coin_id_change_protects`)
- [x] Re-registered validators protected from replay (`vv_req_sec_006_replay_attack_different_announcement`, `vv_req_sec_006_full_replay_scenario`)
- [x] Puzzle hardcodes non-membership byte (`vv_req_sec_006_puzzle_hardcodes_nonmembership`)

## Implementation Notes

- **Announcement:** `puzzles/checkpoint_inner.clsp`
- **Assertion:** `puzzles/registration_coin.clsp`
- **Epoch tracking:** Checkpoint singleton state

### Announcement Generation

```rust
pub fn membership_announcement(
    epoch: u64,
    pubkey: &PublicKey,
    is_member: bool,
) -> [u8; 32] {
    let mut msg = Vec::new();
    msg.extend_from_slice(b"membership");
    msg.extend_from_slice(&epoch.to_be_bytes());  // Epoch included
    msg.extend_from_slice(&pubkey.to_bytes());
    msg.push(if is_member { 1 } else { 0 });
    sha256(&msg)
}
```

## Verification

1. Exit validator at epoch N
2. Re-register same validator at epoch N+1
3. Attempt replay of epoch N non-membership proof
4. Verify replay fails due to coin ID mismatch
5. Generate new non-membership proof at epoch N+2
6. Verify new proof works correctly
