# SEC-011 — Phantom Majority Forgery Resistance

> **Authoritative requirement:** [SEC-011](../NORMATIVE.md#SEC-011)
> **Verification:** [VERIFICATION.md](../VERIFICATION.md)
> **Tracking:** [TRACKING.yaml](../TRACKING.yaml)
> **CHIP reference:** [chip-groth16-l2-consensus.md](../../../../resources/chip-groth16-l2-consensus.md) — Security / Circuit Statement

## Severity

**CRITICAL** — Complete checkpoint forgery by any party with the proving key.

## Summary

The Groth16 circuit's CIR-003 constraint (aggregate key binding) is not
implemented. The `agg_signers` public input is allocated but has NO circuit
constraint tying it to any private witness (signing pubkeys). This allows
an attacker who possesses the proving key and a single BLS keypair to generate
a valid Groth16 proof claiming an arbitrary number of phantom signers, forge
a BLS signature with their single key, and submit a forged checkpoint that
the on-chain CLVM puzzle accepts.

## Attack Scenario

### Prerequisites

- **Proving key**: Published as part of the trusted setup (public artifact)
- **One BLS keypair**: Attacker generates their own; need not be a registered validator
- **Network parameters**: Public (merkle root, validator count, epoch from checkpoint singleton)

### Attack Steps

```
1. Attacker reads current checkpoint state from L1:
   - epoch = N
   - validator_merkle_root = R
   - validator_count = 100

2. Attacker chooses malicious state:
   - new_state_root = 0xATTACKER_STATE (drains L2 funds)

3. Attacker constructs checkpoint_message:
   msg = sha256(attacker_state || R || 100_be8 || (N+1)_be8 || network_id)

4. Attacker sets agg_signers = their_own_pubkey (NOT an aggregate)

5. Attacker generates Groth16 proof with ConsensusCircuit:
   - Public inputs: R, 100, R, 100, their_pubkey, msg
   - Private witness: actual_signers = 51 (PHANTOM)
   - Circuit checks: 2*51 > 100 ✓ (CIR-004 passes)
   - Circuit does NOT check: sum(pubkeys) == agg_signers (CIR-003 absent)
   - Proof generation SUCCEEDS

6. Attacker signs: sig = BLS_Sign(their_sk, msg) using augmented scheme

7. Attacker submits spend bundle:
   - Groth16 proof: valid (pairing check passes)
   - BLS signature: valid (signed with matching key)
   - Checkpoint singleton: accepts the spend, advances epoch
   - Malicious state_root committed on L1
```

### Why Both Checks Pass

| Check | What it verifies | Why it passes for attacker |
|-------|-----------------|---------------------------|
| `bls_pairing_identity` | Groth16 proof equation | Valid proof generated with PK |
| `bls_verify(sig, agg_signers, msg)` | Signature over message | Attacker signed with their own key |

The two-check design assumes CIR-003 binds `agg_signers` to k legitimate
pubkeys. Without CIR-003, `agg_signers` is unconstrained — the attacker
sets it to any key they control.

## Root Cause

In `src/prover/circuit.rs`, the `generate_constraints()` function:

```rust
// Line 501:
// CIR-003: Aggregate key — Phase 3 (non-native G1 arithmetic)

Ok(())
```

The comment is the only reference to CIR-003. No `enforce_equal` or any
other constraint ties `agg_signers` (public input #5) to the private
witness pubkeys.

## Current Mitigations

**None for production deployment.** The DESIGN_DECISIONS.md (Decision 2,
Option C) acknowledges this gap with the note: "Security relies on: the
prover who knows the proving key IS the checkpoint operator, and they have
no incentive to fake the aggregate."

This assumption is INVALID for production because:
1. The proving key is a public artifact (all checkpoint operators need it)
2. The CHIP specifies permissionless checkpoint submission
3. Any party downloading the proving key can forge checkpoints

## Required Fix

CIR-003 MUST be implemented before production deployment. The circuit must
enforce:

```
pk₁ + pk₂ + ... + pkₖ = agg_signers
```

Where each pkᵢ is a private witness with a valid Merkle inclusion proof
(CIR-002) against `validator_merkle_root`.

### Implementation Options (from DESIGN_DECISIONS.md)

1. **Non-native G1 arithmetic** — standard approach, ~50K constraints per addition
2. **Scalar commitment scheme** — hash commitment instead of G1 sum (cheaper, weaker)
3. **Hybrid approach** — Merkle proof in circuit, G1 aggregation via BLS multi-sig outside

## Test Evidence

File: `tests/vv_req_sec_011.rs` — 7 tests, all pass (demonstrating vulnerability):

| Test | What it proves |
|------|---------------|
| `phantom_majority_proof_generation_succeeds` | Proof with k=3, n=5 using single key succeeds |
| `phantom_majority_extreme_ratio` | Proof with k=51, n=100 using single key succeeds |
| `actual_signers_is_unconstrained` | Same agg_signers works with k=6,7,8,9,10 |
| `minority_still_rejected_by_cir_004` | CIR-004 correctly rejects k=2, n=5 |
| `full_forgery_accepted_by_clvm` | **CLVM puzzle accepts forged checkpoint end-to-end** |
| `cir_003_not_enforced_in_circuit` | Source confirms no constraint after CIR-003 comment |
| `merkle_root_is_private_witness` | Poseidon root is private, not public input |

## Acceptance Criteria (for fix)

- [x] CIR-003 constraint enforced: G1 sum of witness pubkeys == agg_signers (`src/prover/g1_gadget.rs`)
- [x] Phantom majority proof generation FAILS with secure constructor (`vv_req_sec_011_phantom_majority_proof_generation_fails`)
- [x] Extreme phantom ratio blocked (`vv_req_sec_011_phantom_majority_extreme_ratio_blocked`)
- [x] Signers constrained to aggregate key (`vv_req_sec_011_actual_signers_constrained_to_keys`)
- [x] Legitimate majority proofs still succeed (`vv_req_sec_011_legitimate_majority_succeeds`)
- [x] Constructor accepts signing_pubkeys for CIR-003 activation (`vv_req_sec_011_constructor_accepts_signing_pubkeys`)

## Impact Assessment

| Aspect | Assessment |
|--------|------------|
| **Exploitability** | Trivial — requires only public proving key + one BLS keypair |
| **Impact** | Complete — arbitrary L2 state can be committed to L1 |
| **Affected** | All deployments using current circuit |
| **Detection** | None on-chain — forged checkpoint looks identical to legitimate one |
| **Recovery** | Redeploy checkpoint singleton with fixed circuit + new trusted setup |
