## Summary

The Groth16 circuit proves a precise statement about validator set membership and signing majority without revealing which specific validators signed. This zero-knowledge property allows efficient on-chain verification at constant cost regardless of validator set size.

## Specification

### Circuit Statement (Informal)

> "I know k BLS pubkeys, each with a valid Merkle inclusion proof against `validator_merkle_root`, whose G1 sum equals `agg_signers`, and where 2k > `validator_count`."

### Circuit Statement (Formal)

Given public inputs:
- `validator_merkle_root`: H (32 bytes)
- `validator_count`: n (integer)
- `agg_signers`: A (G1 point)

The prover demonstrates knowledge of private witnesses:
- `pubkeys`: [pk₁, pk₂, ..., pkₖ] (k BLS pubkeys)
- `proofs`: [π₁, π₂, ..., πₖ] (k Merkle proofs)

Such that:
1. ∀i ∈ [1,k]: `verify_merkle(pkᵢ, πᵢ, validator_merkle_root) = true`
2. `pk₁ + pk₂ + ... + pkₖ = agg_signers` (G1 addition)
3. `2k > validator_count`

### What the Circuit Does NOT Prove

The circuit deliberately excludes:
- **BLS signature validity**: Handled by `bls_verify` on-chain
- **Checkpoint message commitment**: Part of public inputs, not verified inside circuit
- **New state validity**: Attested by majority signature

### Why This Design

**BLS inside BLS12-381 is expensive:**
Verifying BLS12-381 signatures inside a BLS12-381 Groth16 circuit requires emulating field arithmetic, adding ~2M constraints per signature. The two-check design avoids this.

**Separation of concerns:**
- Circuit: "These signers are legitimate and form a majority"
- bls_verify: "These signers actually signed this message"

### Private Witnesses

| Witness | Type | Count | Description |
|---------|------|-------|-------------|
| `pubkeys` | G1 (48 bytes each) | k | Signing validators' pubkeys |
| `merkle_proofs` | Vec<H> | k × TREE_DEPTH | Inclusion proofs |
| `leaf_indices` | u64 | k | Slot positions in tree |

### Proof Generation Flow

```rust
fn generate_proof(
    merkle_root: Bytes32,
    count: u64,
    signers: &[PublicKey],
    proofs: &[MerkleProof],
) -> Groth16Proof {
    // 1. Allocate public inputs
    let pi = allocate_public_inputs(merkle_root, count, ...);

    // 2. Allocate private witnesses
    let witnesses = allocate_witnesses(signers, proofs);

    // 3. Build constraint system
    let cs = build_constraints(pi, witnesses);

    // 4. Generate proof
    arkworks::prove(proving_key, cs)
}
```

### Constraint Breakdown

| Constraint Type | Per Signer | Total (k=64, depth=32) |
|----------------|------------|------------------------|
| SHA-256 (pubkey→leaf) | ~25,000 | ~1,600,000 |
| Merkle path verification | ~25,000 | ~1,600,000 |
| G1 decompression | ~50,000 | ~3,200,000 |
| G1 addition | ~10,000 | ~640,000 |
| Majority check | - | ~1,000 |
| **Total** | | **~7,041,000** |

## Acceptance Criteria

- [ ] Circuit proves all three components of the statement
- [ ] Valid witnesses produce verifying proof
- [ ] Invalid witnesses (wrong Merkle proof) fail proof generation
- [ ] Invalid witnesses (wrong aggregate) fail proof generation
- [ ] Minority signers fail proof generation
- [ ] Proof verifies on-chain via checkpoint singleton

## Verification

1. Generate valid proof with majority signers
2. Verify proof on-chain → success
3. Generate proof with non-member pubkey → proof generation fails
4. Generate proof with wrong aggregate → proof generation fails
5. Generate proof with minority → proof generation fails
6. Fuzz test: random valid inputs → all proofs verify
