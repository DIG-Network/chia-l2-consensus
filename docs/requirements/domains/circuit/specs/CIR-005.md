# CIR-005 — Public Inputs

> **Authoritative requirement:** [CIR-005](../NORMATIVE.md#CIR-005)
> **Verification:** [VERIFICATION.md](../VERIFICATION.md)
> **Tracking:** [TRACKING.yaml](../TRACKING.yaml)

## Summary

The circuit accepts exactly 6 public inputs in a fixed order. These inputs are visible to the verifier and are used to compute the VK input point for Groth16 verification. The order must match between circuit, proving key, and verification key.

## Specification

### Public Input Order

| Index | Name | Type | Size | Description |
|-------|------|------|------|-------------|
| 1 | `validator_merkle_root` | Bytes32 | 32 bytes | Current Merkle root from singleton |
| 2 | `validator_count` | u64 | 8 bytes | Current validator count |
| 3 | `new_validator_merkle_root` | Bytes32 | 32 bytes | New Merkle root for checkpoint |
| 4 | `new_validator_count` | u64 | 8 bytes | New validator count |
| 5 | `agg_signers` | G1 | 48 bytes | Aggregate of signing pubkeys |
| 6 | `checkpoint_message` | Bytes32 | 32 bytes | Hash of checkpoint content |

### IC Point Correspondence

The verification key has 7 IC points (index 0 is constant term):

| IC Index | Public Input Index | Input |
|----------|-------------------|-------|
| 0 | - | Constant term |
| 1 | 1 | validator_merkle_root |
| 2 | 2 | validator_count |
| 3 | 3 | new_validator_merkle_root |
| 4 | 4 | new_validator_count |
| 5 | 5 | agg_signers |
| 6 | 6 | checkpoint_message |

### VK Input Computation

```
vk_input = IC[0]
         + scalar(validator_merkle_root) × IC[1]
         + scalar(validator_count) × IC[2]
         + scalar(new_validator_merkle_root) × IC[3]
         + scalar(new_validator_count) × IC[4]
         + scalar(agg_signers) × IC[5]
         + scalar(checkpoint_message) × IC[6]
```

### scalar() Function

```
scalar(bytes) = sha256(bytes) as big-endian u256, mod r
```

Where r is the BLS12-381 scalar field order:
```
r = 0x73eda753299d7d483339d80809a1d80553bda402fffe5bfeffffffff00000001
```

### Why These Public Inputs

**validator_merkle_root (current):**
- Binds proof to current validator set
- Prevents proofs generated against old/future sets

**validator_count (current):**
- Used in majority check (2k > count)
- Must match on-chain state

**new_validator_merkle_root:**
- Committed in checkpoint message
- Attested by majority signature

**new_validator_count:**
- Committed in checkpoint message
- Used for next checkpoint's majority check

**agg_signers:**
- Binds ZK proof to BLS signature
- Verified to be sum of k pubkeys

**checkpoint_message:**
- The message that was signed
- Commits to all new state values

### Order Sensitivity

**If order is wrong:**
- IC points multiply wrong scalar values
- VK input point is incorrect
- Pairing check fails
- All proofs rejected

**This is a silent failure** — no error message indicates which input is wrong.

## Acceptance Criteria

- [ ] Exactly 6 public inputs allocated
- [ ] Order matches specification
- [ ] Order matches VK IC point assignment
- [ ] scalar() applied correctly to each input
- [ ] Wrong order → verification fails
- [ ] Missing input → proof generation fails

## Implementation Notes

- **Circuit definition:** `src/circuit/mod.rs`
- **Public input allocation:** Must use `new_input()` in correct order
- **Testing:** Cross-check with checkpoint singleton's VK input computation

### Common Mistakes

1. **Allocating in wrong order**: Inputs must be `new_input()` in exact sequence
2. **Using witness instead of input**: `new_witness()` makes it private
3. **Scalar computation mismatch**: Circuit and verifier must use identical `scalar()`

## Verification

1. Generate proof with all inputs in correct order
2. Verify on-chain → success
3. Swap inputs 1 and 2 in proof → failure
4. Omit one input → proof generation fails
5. Add extra input → proof generation fails or wrong output
