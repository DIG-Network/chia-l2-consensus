# CIR-003 — Aggregate Key Constraint

> **Authoritative requirement:** [CIR-003](../NORMATIVE.md#CIR-003)
> **Verification:** [VERIFICATION.md](../VERIFICATION.md)
> **Tracking:** [TRACKING.yaml](../TRACKING.yaml)
> **CHIP reference:** [chip-groth16-l2-consensus.md](../../../../resources/chip-groth16-l2-consensus.md) — Circuit Statement

## Summary

The circuit verifies that the G1 sum of the k signing pubkeys equals the public input `agg_signers`. This binds the ZK proof to the specific aggregate public key that will be used for BLS signature verification on-chain.

## Specification

### Constraint Definition

```
pk₁ + pk₂ + ... + pkₖ = agg_signers
```

Where:
- `pkᵢ` are G1 points (BLS pubkeys) — private witnesses
- `agg_signers` is a G1 point — public input
- Addition is elliptic curve point addition on BLS12-381 G1

### G1 Addition in Circuit

```rust
fn aggregate_pubkeys(
    cs: ConstraintSystemRef<Fr>,
    pubkeys: &[G1Var],
    expected_aggregate: &G1Var,
) -> Result<(), SynthesisError> {
    // Start with identity (point at infinity)
    let mut aggregate = G1Var::zero();

    for pk in pubkeys {
        // G1 addition: ~10,000 constraints per addition
        aggregate = aggregate.add(pk)?;
    }

    // Verify sum equals expected
    aggregate.enforce_equal(expected_aggregate)?;

    Ok(())
}
```

### G1 Point Representation in Circuit

BLS12-381 G1 points in the circuit:
- Affine coordinates (x, y) over Fq
- Each coordinate: 381 bits ≈ 6 limbs of 64 bits
- Point addition uses affine formulas

### Why This Constraint Matters

Without this constraint:
1. Attacker claims `agg_signers = pk_attacker`
2. Attacker provides Merkle proofs for k legitimate validators
3. Attacker signs checkpoint with their own key
4. On-chain: `bls_verify(attacker_sig, pk_attacker, msg)` passes
5. Attack succeeds despite not controlling any real validator keys

With this constraint:
1. Attacker must provide k pubkeys that SUM to `agg_signers`
2. Those k pubkeys must all be in the Merkle tree
3. Attacker cannot make k legitimate keys sum to their key
4. Attack fails

### Handling Variable k

The circuit is compiled for MAX_SIGNERS. If actual k < MAX_SIGNERS:

**Option 1: Zero padding**
```rust
// Unused slots contribute identity point (zero)
for i in k..MAX_SIGNERS {
    pubkeys[i] = G1::identity();
}
// identity + pk = pk, so sum is correct
```

**Option 2: Duplicate valid keys**
```rust
// Unused slots duplicate existing keys with negation
for i in k..MAX_SIGNERS {
    pubkeys[i] = pubkeys[0];        // Duplicate
    pubkeys[i+1] = -pubkeys[0];     // Negate
}
// pk + (-pk) = 0, so sum is correct
```

Both approaches must maintain valid Merkle proofs for the padding entries.

### Constraint Count

| Operation | Constraints |
|-----------|-------------|
| G1 decompression (per key) | ~50,000 |
| G1 addition (per pair) | ~10,000 |
| Equality check | ~3,000 |
| **Total (k=64)** | **~3,840,000** |

### Soundness

The circuit ensures:
- Only pubkeys with valid Merkle proofs contribute to sum
- The exact k pubkeys that form the aggregate are verified members
- No "phantom" pubkeys can be added to inflate the count

## Acceptance Criteria

- [ ] G1 sum of k pubkeys equals agg_signers public input
- [ ] Sum uses proper elliptic curve addition
- [ ] Identity/zero handling for k < MAX_SIGNERS
- [ ] Wrong aggregate → proof generation fails
- [ ] Single pubkey test: k=1, agg_signers = pk₁
- [ ] Subtraction attack: can't use pk - pk' to manipulate

## Implementation Notes

- **Codebase:** `src/circuit/aggregate.rs`
- **G1 gadget:** `ark-ec` or custom BLS12-381 implementation
- **Optimization:** Montgomery ladder for additions

### Arkworks Implementation

```rust
use ark_bls12_381::{G1Projective, G1Affine};
use ark_r1cs_std::groups::CurveVar;

fn enforce_aggregation(
    pubkeys: &[G1Var],
    agg_signers: &G1Var,
) -> Result<(), SynthesisError> {
    let mut sum = G1Var::zero();
    for pk in pubkeys.iter() {
        sum = sum + pk;
    }
    sum.enforce_equal(agg_signers)
}
```

## Verification

1. k=3 signers, compute correct aggregate → proof succeeds
2. k=3 signers, wrong aggregate (off by one) → proof fails
3. k=1 signer, aggregate = single key → proof succeeds
4. k=64 signers (max) → proof succeeds
5. Pad with zeros for k < MAX_SIGNERS → sum still correct
