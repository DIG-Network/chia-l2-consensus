# CIR-006 — Circuit Parameters

> **Authoritative requirement:** [CIR-006](../NORMATIVE.md#CIR-006)
> **Verification:** [VERIFICATION.md](../VERIFICATION.md)
> **Tracking:** [TRACKING.yaml](../TRACKING.yaml)

## Summary

The circuit is parameterized by two values fixed at trusted setup time: `MAX_SIGNERS` (maximum number of simultaneous signers the circuit can verify) and `TREE_DEPTH` (depth of the sparse Merkle tree). These parameters cannot be changed without a new trusted setup ceremony.

## Specification

### Parameters

| Parameter | Type | Typical Value | Description |
|-----------|------|---------------|-------------|
| `MAX_SIGNERS` | usize | 64-256 | Max validators that can sign a single checkpoint |
| `TREE_DEPTH` | u32 | 32 | Depth of sparse Merkle tree (supports 2^32 slots) |

### MAX_SIGNERS

**What it controls:**
- Size of the pubkey array in the circuit
- Number of Merkle proofs verified
- Total constraint count
- Proof generation time

**Trade-offs:**

| MAX_SIGNERS | Constraints (approx) | Proof Time | Use Case |
|-------------|---------------------|------------|----------|
| 16 | ~14M | ~2 min | Small testnets |
| 64 | ~56M | ~8 min | Medium networks |
| 256 | ~224M | ~30 min | Large networks |

**Relationship to validator_count:**
- `MAX_SIGNERS` is the circuit's capacity
- `validator_count` is the current number of validators
- `k` is actual signers for this checkpoint
- Constraint: `k ≤ MAX_SIGNERS`
- If `validator_count > 2 × MAX_SIGNERS`, majority is impossible with this circuit

### TREE_DEPTH

**What it controls:**
- Number of Merkle proof levels
- Constraints per signer (~25,000 SHA-256 per level)
- Maximum validator slots: 2^TREE_DEPTH

**Typical value: 32**
- Supports 2^32 ≈ 4.3 billion validator slots
- More than enough for any practical validator set
- 32 SHA-256 operations per proof verification

**Why not deeper?**
- Each additional level adds ~25,000 constraints per signer
- Diminishing returns beyond 2^32 slots
- Proof generation becomes slower

### Parameter Binding

Parameters are embedded in the circuit at compile time:

```rust
pub struct ConsensusCircuit<const MAX_SIGNERS: usize, const TREE_DEPTH: usize> {
    // ... fields
}

// Instantiate specific circuit
type ProductionCircuit = ConsensusCircuit<64, 32>;
```

### Trusted Setup Binding

The proving key (PK) and verification key (VK) are generated FOR a specific circuit:

```
Circuit(MAX_SIGNERS=64, TREE_DEPTH=32) → (PK, VK)
```

A VK generated for MAX_SIGNERS=64 cannot verify proofs from a circuit with MAX_SIGNERS=128.

### Coordination Requirements

**All components must agree:**

| Component | Must Match |
|-----------|-----------|
| Circuit source | MAX_SIGNERS, TREE_DEPTH |
| Proving key | Generated from matching circuit |
| Verification key | Generated from matching circuit |
| Checkpoint singleton | TREE_DEPTH curried in |
| SMT implementation | TREE_DEPTH |
| Network config | MAX_SIGNERS, TREE_DEPTH |

### Upgrading Parameters

**Cannot upgrade in place.** Required steps:

1. Design new circuit with new parameters
2. Run new trusted setup ceremony
3. Deploy new checkpoint singleton with new VK
4. Migrate validators to new system
5. Old system continues until migration complete

### Choosing Parameters

**MAX_SIGNERS considerations:**
- Must be ≥ expected maximum validator participation
- Higher = slower proofs
- Can set higher than expected and pad with zeros

**TREE_DEPTH considerations:**
- 32 is standard (matches CHIP recommendation)
- Must match SMT implementation exactly
- Rarely needs to change

## Acceptance Criteria

- [ ] MAX_SIGNERS is compile-time constant
- [ ] TREE_DEPTH is compile-time constant
- [ ] VK is bound to specific parameter values
- [ ] Checkpoint singleton TREE_DEPTH matches circuit
- [ ] k ≤ MAX_SIGNERS enforced
- [ ] Merkle proofs have exactly TREE_DEPTH siblings
- [ ] Network config documents both parameters

## Implementation Notes

- **Circuit definition:** `src/circuit/mod.rs` with const generics
- **Build time:** Parameters selected before `cargo build`
- **Config file:** `network_config.json` includes both parameters

### Compile-Time Configuration

```rust
// In circuit/mod.rs
pub const MAX_SIGNERS: usize = 64;
pub const TREE_DEPTH: usize = 32;

// Or via build feature
#[cfg(feature = "large_circuit")]
pub const MAX_SIGNERS: usize = 256;
```

## Verification

1. Compile circuit with MAX_SIGNERS=64
2. Generate VK
3. Deploy checkpoint singleton with that VK
4. Verify TREE_DEPTH=32 in all components
5. Try proof with k=65 signers → proof generation fails
6. Try proof with depth-31 proofs → verification fails
