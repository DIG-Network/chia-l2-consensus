# CIR-002 — Merkle Membership Constraint

> **Authoritative requirement:** [CIR-002](../NORMATIVE.md#CIR-002)
> **Verification:** [VERIFICATION.md](../VERIFICATION.md)
> **Tracking:** [TRACKING.yaml](../TRACKING.yaml)
> **CHIP reference:** [chip-groth16-l2-consensus.md](../../../../resources/chip-groth16-l2-consensus.md) — Circuit Statement

## Summary

For each of the k signing pubkeys, the circuit verifies a Merkle inclusion proof demonstrating the pubkey exists in the validator set. This constraint ensures only registered validators can be counted toward the signing majority.

## Specification

### Constraint Definition

For each signer i ∈ [1, k]:
```
verify_merkle_path(sha256(pubkeyᵢ), proofᵢ, indexᵢ, TREE_DEPTH) = validator_merkle_root
```

### Merkle Verification in Circuit

```rust
fn verify_merkle_membership(
    cs: ConstraintSystemRef<Fr>,
    pubkey: &G1Var,
    siblings: &[FpVar],
    index: &FpVar,
    expected_root: &FpVar,
) -> Result<(), SynthesisError> {
    // 1. Compute leaf = sha256(pubkey)
    let leaf = sha256_gadget(cs.clone(), pubkey.to_bytes())?;

    // 2. Walk up the tree
    let mut current = leaf;
    for (level, sibling) in siblings.iter().enumerate() {
        let bit = index.to_bits_le()?[level].clone();

        // If bit=0: hash(current, sibling)
        // If bit=1: hash(sibling, current)
        let left = FpVar::conditionally_select(&bit, sibling, &current)?;
        let right = FpVar::conditionally_select(&bit, &current, sibling)?;

        current = sha256_gadget(cs.clone(), &[left, right])?;
    }

    // 3. Verify root matches
    current.enforce_equal(expected_root)?;

    Ok(())
}
```

### SHA-256 in Circuit

SHA-256 is implemented using R1CS constraints:
- Each SHA-256 round: ~1,000 constraints
- 64 rounds + padding: ~25,000 constraints total
- Applied once per pubkey (leaf computation)
- Applied TREE_DEPTH times per signer (path verification)

### Leaf Computation

The leaf value for an active validator:
```
leaf = sha256(pubkey)  // pubkey is 48 bytes compressed G1
```

This matches the off-chain SMT implementation ([SMT-002](../../smt/specs/SMT-002.md)).

### Index Computation

The Merkle tree index (slot) is derived from pubkey:
```
slot = first_8_bytes_be(sha256(pubkey)) mod 2^TREE_DEPTH
```

The circuit receives the index as a private witness and verifies the path.

### Sibling Ordering

**Critical for cross-implementation compatibility:**
- At each level, if index bit = 0: current is LEFT child
- At each level, if index bit = 1: current is RIGHT child
- Hash is always `sha256(left || right)`

### Per-Signer Constraint Count

| Operation | Constraints |
|-----------|-------------|
| G1 → bytes conversion | ~5,000 |
| SHA-256 (leaf) | ~25,000 |
| SHA-256 × TREE_DEPTH (path) | ~800,000 |
| Bit decomposition | ~256 |
| Conditional selects | ~64 |
| **Total per signer** | **~830,000** |

### Handling Fewer Than MAX_SIGNERS

If k < MAX_SIGNERS, the remaining slots use dummy witnesses:
- Dummy pubkey: Valid pubkey from the tree (can reuse a real one)
- Dummy proof: Valid proof for that pubkey
- These are valid witnesses that don't affect the aggregate computation because they're zeroed out in the aggregation constraint

## Acceptance Criteria

- [ ] Each signer's pubkey has Merkle proof verified
- [ ] Leaf computed as sha256(pubkey)
- [ ] Path verification uses correct sibling ordering
- [ ] All k proofs must verify against same root
- [ ] Invalid proof → proof generation fails
- [ ] Root mismatch → proof generation fails

## Implementation Notes

- **Codebase:** `src/circuit/merkle.rs`
- **SHA-256 gadget:** `ark-crypto-primitives` or custom implementation
- **Optimization:** Batched SHA-256 for shared constants

### Common Bugs

1. **Sibling ordering mismatch**: Rust and circuit must agree on left-first
2. **Index interpretation**: Big-endian vs little-endian bits
3. **Padding differences**: SHA-256 padding must match exactly

## Verification

1. Generate proof with valid Merkle proofs → success
2. Corrupt one sibling hash → proof generation fails
3. Use proof for wrong pubkey → proof generation fails
4. Use proof against wrong root → proof generation fails
5. Cross-impl test: Rust-computed root = circuit-computed root
