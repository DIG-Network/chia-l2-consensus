# CIR-004 — Majority Threshold Constraint

> **Authoritative requirement:** [CIR-004](../NORMATIVE.md#CIR-004)
> **Verification:** [VERIFICATION.md](../VERIFICATION.md)
> **Tracking:** [TRACKING.yaml](../TRACKING.yaml)

## Summary

The circuit enforces that the number of signing validators k forms a strict majority of the validator set: `2k > validator_count`. This prevents minority groups from submitting fraudulent checkpoints.

## Specification

### Constraint Definition

```
2k > validator_count
```

Equivalently:
```
2k ≥ validator_count + 1
2k - validator_count - 1 ≥ 0
```

### Why Strict Majority (>)

**Not ≥ (at least half):**
- With 100 validators, k=50 would pass
- A 50-50 split could result in conflicting checkpoints
- No clear "winner" in case of disagreement

**Strict majority (>):**
- With 100 validators, k=51 is minimum
- Only one group can have >50%
- Guarantees unique consensus

### Implementation in Circuit

```rust
fn enforce_majority(
    cs: ConstraintSystemRef<Fr>,
    k: &FpVar,  // Actual count of signers
    validator_count: &FpVar,  // Public input
) -> Result<(), SynthesisError> {
    // Compute 2k
    let two_k = k.double()?;

    // Compute 2k - validator_count - 1
    let one = FpVar::one();
    let diff = two_k - validator_count - one;

    // Assert diff >= 0 (non-negative)
    // In a prime field, this means diff is "small" (fits in expected bits)
    diff.enforce_cmp(&FpVar::zero(), Ordering::Greater, true)?;

    Ok(())
}
```

### Range Constraints

In a prime field, "non-negative" requires range checks:
- `validator_count` must be < 2^64 (realistic validator counts)
- `k` must be ≤ MAX_SIGNERS (circuit parameter)
- `2k - validator_count - 1` must be in valid range

### k Derivation

The circuit doesn't receive k directly. Instead:
- k = number of non-zero entries in the pubkey array
- Or k is derived from which slots have real vs padding witnesses

```rust
// Count actual signers (non-identity pubkeys)
let mut k = FpVar::zero();
for pk in pubkeys.iter() {
    let is_real = pk.is_zero()?.not();  // Boolean
    k = k + is_real.select(&one, &zero)?;
}
```

### Edge Cases

| validator_count | Minimum k | 2k | Valid? |
|-----------------|-----------|-----|--------|
| 1 | 1 | 2 | ✓ (2 > 1) |
| 2 | 2 | 4 | ✓ (4 > 2) |
| 3 | 2 | 4 | ✓ (4 > 3) |
| 100 | 51 | 102 | ✓ (102 > 100) |
| 100 | 50 | 100 | ✗ (100 > 100 false) |
| 0 | 1 | 2 | ✓ (technically, but count=0 is degenerate) |

### Constraint Count

| Operation | Constraints |
|-----------|-------------|
| k computation (MAX_SIGNERS comparisons) | ~MAX_SIGNERS × 10 |
| Multiplication by 2 | ~1 |
| Subtraction | ~1 |
| Range check (non-negative) | ~256 |
| **Total** | **~1,000** |

This is negligible compared to Merkle and aggregation constraints.

### Security: Why Count is Public Input

`validator_count` comes from the checkpoint singleton state, not the prover:
- Prover can't lie about total validator count
- On-chain verification uses the same count
- Prevents "I have majority of my fake small set" attack

## Verification

1. k=51, count=100 → proof generation succeeds
2. k=50, count=100 → proof generation fails
3. k=1, count=1 → proof generation succeeds
4. k=2, count=3 → proof generation succeeds
5. k=1, count=3 → proof generation fails
6. k=0 (no signers) → proof generation fails
