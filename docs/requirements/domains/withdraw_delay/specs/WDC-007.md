# WDC-007 — Permissionless Release

> **Authoritative requirement:** [WDC-007](../NORMATIVE.md#WDC-007)
> **Verification:** [VERIFICATION.md](../VERIFICATION.md)
> **Tracking:** [TRACKING.yaml](../TRACKING.yaml)
> **Spec reference:** [spec-withdraw-delay-coin.md](../../../../resources/spec-withdraw-delay-coin.md) — Security Properties

## Summary

The withdraw delay coin emits no `AGG_SIG_ME` or `AGG_SIG_UNSAFE` conditions.
After the delay period, anyone can spend it. Since funds can only go to the
curried `DESTINATION`, this is safe and ensures validators don't need to be
online at the exact moment the delay expires.

## Specification

### No Signature Conditions

The puzzle output contains exactly:
1. `ASSERT_HEIGHT_RELATIVE(WITHDRAW_DELAY_BLOCKS)`
2. `CREATE_COIN(DESTINATION, AMOUNT)`

No AGG_SIG of any kind. The spend bundle uses `G2Affine::identity()` as the
aggregated signature.

### Why Permissionless Is Safe

| Concern | Mitigation |
|---------|------------|
| Someone steals funds | Impossible — funds go to curried DESTINATION only |
| Someone changes destination | Impossible — DESTINATION is curried, not in solution |
| Someone changes amount | Impossible — AMOUNT is curried, not in solution |
| Someone spends before delay | Impossible — ASSERT_HEIGHT_RELATIVE enforced by consensus |
| Griefing (spam spending) | Harmless — funds go to the correct destination |

### Who Can Spend

| Caller | Can spend? | Effect |
|--------|-----------|--------|
| Exiting validator | Yes | Funds go to their DESTINATION |
| Other validators | Yes | Funds still go to DESTINATION |
| Third-party service | Yes | Can "help" release funds |
| Malicious actor | Yes | Cannot redirect funds |

### Operational Benefit

Permissionless release means:
- Validators can go offline after initiating recovery
- An automated service can release delay coins on behalf of all exiting validators
- No key management is needed for the release step

## Acceptance Criteria

- [ ] CLVM output has no AGG_SIG_ME (opcode 49)
- [ ] CLVM output has no AGG_SIG_UNSAFE (opcode 50)
- [ ] Spend bundle uses identity signature
- [ ] Third-party can successfully spend after delay
- [ ] Funds always go to curried DESTINATION regardless of who spends
