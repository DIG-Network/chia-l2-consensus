# WDC-002 — Time Lock Enforcement

> **Authoritative requirement:** [WDC-002](../NORMATIVE.md#WDC-002)
> **Verification:** [VERIFICATION.md](../VERIFICATION.md)
> **Tracking:** [TRACKING.yaml](../TRACKING.yaml)
> **Spec reference:** [spec-withdraw-delay-coin.md](../../../../resources/spec-withdraw-delay-coin.md)

## Summary

The withdraw delay coin uses Chia's `ASSERT_HEIGHT_RELATIVE` condition to
enforce a mandatory waiting period. This condition is verified by the Chia
consensus layer, not by custom CLVM logic, making it unforgeable and
unbypassable.

## Specification

### ASSERT_HEIGHT_RELATIVE Semantics

`ASSERT_HEIGHT_RELATIVE(N)` means: this spend is only valid if the current
block height is at least N blocks higher than the block in which this coin
was created.

- **Condition code:** 82 (0x52)
- **Argument:** Number of blocks (non-negative integer)
- **Enforced by:** Chia full node consensus, not CLVM execution
- **Behavior on failure:** Spend is rejected by the mempool/consensus

### Timing Properties

| Property | Value |
|----------|-------|
| Timer starts | When the withdraw delay coin is created (registration coin spend confirmed) |
| Timer unit | L1 blocks (~18.75 seconds per block on mainnet) |
| Timer check | At inclusion time, not at mempool submission time |
| Precision | Block-level (not sub-block) |

### Example Delays

| `WITHDRAW_DELAY_BLOCKS` | Approximate Wait |
|--------------------------|------------------|
| 10 | ~3 minutes |
| 32 | ~10 minutes |
| 256 | ~80 minutes |
| 1024 | ~5.3 hours |
| 4608 | ~24 hours |

### Interaction with Mempool

A spend bundle containing `ASSERT_HEIGHT_RELATIVE(N)` where N blocks have
not yet passed will be:
- Rejected by the mempool if `current_height - coin_creation_height < N`
- Accepted by the mempool if the node estimates it will be valid soon
- Always validated at block inclusion time by consensus

### Cannot Be Bypassed

The delay cannot be shortened because:
1. `WITHDRAW_DELAY_BLOCKS` is curried in, not in the solution
2. No alternative spend path exists
3. `ASSERT_HEIGHT_RELATIVE` is enforced by the Chia node, not CLVM
4. No passthrough conditions can override it

## Acceptance Criteria

- [ ] CLVM output contains ASSERT_HEIGHT_RELATIVE (opcode 82)
- [ ] Delay value matches curried WITHDRAW_DELAY_BLOCKS
- [ ] Simulator rejects spend before delay period has elapsed
- [ ] Simulator accepts spend after delay period has elapsed
- [ ] Delay value of 0 allows immediate spend
- [ ] Delay value of 1 requires at least 1 block to pass
- [ ] Different delay values produce different puzzle hashes

## Implementation Notes

- **CLVM condition code:** 82 (`ASSERT_HEIGHT_RELATIVE`)
- **Rue syntax:** `AssertHeightRelative(WITHDRAW_DELAY_BLOCKS)`

### Simulator Testing

The chia-sdk-test `Simulator` supports advancing block heights. Use
`sim.advance_to_height(h)` or equivalent to test the delay:

```rust
// Create withdraw delay coin at height H
sim.push_tx(recovery_bundle)?;
let creation_height = sim.height();

// Try to spend before delay — must fail
let release = build_release(delay_coin, ...);
assert!(sim.push_tx(release.clone()).is_err());

// Advance past delay
sim.advance_to_height(creation_height + delay_blocks);

// Now spend succeeds
sim.push_tx(release)?;
```
