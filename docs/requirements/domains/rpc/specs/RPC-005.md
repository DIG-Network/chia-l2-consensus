# RPC-005 — Wallet Integration for Collateral Funding

> **Authoritative requirement:** [RPC-005](../NORMATIVE.md#RPC-005)
> **Verification:** [VERIFICATION.md](../VERIFICATION.md)
> **Tracking:** [TRACKING.yaml](../TRACKING.yaml)

## Summary

When a validator registers, they must fund `COLLATERAL_AMOUNT` mojos. The
crate uses [`dig-l1-wallet`](https://crates.io/crates/dig-l1-wallet) for
coin selection: picking which XCH coins from the validator's wallet to spend
as collateral, computing change, and estimating fees.

## Specification

### Why dig-l1-wallet

The validator needs to provide XCH for collateral. This requires:
1. Knowing which coins the validator owns (wallet state)
2. Selecting optimal coins to meet the collateral amount (coin selection)
3. Computing change (excess returned to validator)
4. Estimating fees for the transaction

`dig-l1-wallet` handles all of these via its `select_coins()` and
`estimate_fee()` methods. It uses `chia-query` internally for blockchain
state, so it shares the same connection infrastructure.

### Registration Flow with Coin Selection

```rust
pub async fn register_validator(
    &self,
    validator_sk: &SecretKey,
    wallet: &L1Wallet,
    wallet_name: &str,
    account_index: u32,
    fee: u64,
) -> ConsensusResult<SpendBundle> {
    let collateral = self.config.collateral_amount;

    // 1. Select coins from validator's wallet to fund collateral + fee
    let selection = wallet
        .select_coins(
            wallet_name,
            Some(account_index),
            collateral + fee,
            CoinSelectionStrategy::LargestFirst,
        )
        .await
        .map_err(|e| ConsensusError::InsufficientFunds(e.to_string()))?;

    // selection.coins — coins to spend
    // selection.total — sum of selected coins
    // selection.change — excess to return

    // 2. Build the network coin spend (creates registration coin)
    let mut ctx = SpendContext::new();
    let network_coin_spend = build_network_coin_spend(
        &mut ctx,
        &self.state()?.network_coin,
        validator_sk,
        &self.config,
    )?;

    // 3. Build funding coin spends (selected XCH → collateral + change)
    let funding_spends = build_funding_spends(
        &mut ctx,
        &selection.coins,
        collateral,
        selection.change,
        validator_change_puzzle_hash,
        fee,
    )?;

    // 4. Combine into single bundle
    let mut coin_spends = vec![network_coin_spend];
    coin_spends.extend(funding_spends);

    Ok(SpendBundle {
        coin_spends,
        aggregated_signature: aggregate_sigs,
    })
}
```

### Coin Selection Strategies

| Strategy | When to Use |
|----------|------------|
| `LargestFirst` | Default for registration — minimizes input count |
| `SmallestFirst` | Consolidate dust (not typical for registration) |
| `Knapsack` | Optimal for exact matches (delegates to chia-wallet-sdk) |

### Fee Estimation

```rust
// Estimate fee for the registration transaction
// Registration = 1 network coin spend + N funding spends + 1 change
let estimated_fee = wallet
    .estimate_fee(2 + selection.coins.len())
    .await
    .unwrap_or(0);
```

### What dig-l1-wallet Does NOT Do

The crate does NOT use dig-l1-wallet for:
- Key generation (validators generate their own BLS keys per VAL-001)
- Transaction broadcasting (per API-008)
- Checkpoint operations (no wallet funds involved)
- Collateral recovery (funds go to specified destination, not back to wallet)

### L1Wallet Lifecycle in ConsensusClient

The `L1Wallet` is NOT stored in `ConsensusClient`. It is passed as a
parameter to `register_validator()` only. This keeps the consensus crate
decoupled from wallet state:

```rust
// The L2 system owns the wallet
let wallet = L1Wallet::new(wallet_config).await?;
wallet.unlock("my_wallet", "password")?;

// Pass wallet to registration only
let bundle = client.register_validator(&sk, &wallet, "my_wallet", 0, fee).await?;

// L2 broadcasts
node.push_tx(&bundle).await?;
```

## Acceptance Criteria

- [ ] `dig-l1-wallet` added as dependency in Cargo.toml
- [ ] `register_validator()` accepts `&L1Wallet` parameter
- [ ] Coin selection uses `select_coins()` with configurable strategy
- [ ] Selected coins included in spend bundle as funding spends
- [ ] Change returned to validator's puzzle hash
- [ ] `InsufficientFunds` error when wallet balance < collateral + fee
- [ ] Fee estimation via `estimate_fee()` available
- [ ] L1Wallet not stored in ConsensusClient (passed per-call)
