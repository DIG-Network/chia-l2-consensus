# RPC-001 — Blockchain Query Client (chia-query)

> **Authoritative requirement:** [RPC-001](../NORMATIVE.md#RPC-001)
> **Verification:** [VERIFICATION.md](../VERIFICATION.md)
> **Tracking:** [TRACKING.yaml](../TRACKING.yaml)

## Summary

The crate uses [`chia-query`](https://crates.io/crates/chia-query) as its
blockchain query backend. `chia-query` connects to Chia peers directly
(decentralized) with automatic fallback to the coinset.org API. This replaces
the need for a custom RPC client or a locally running Chia full node.

## Specification

### Client Initialization

```rust
use chia_query::{ChiaQuery, ChiaQueryConfig, NetworkType};

let config = ChiaQueryConfig {
    network: NetworkType::Mainnet,  // or Testnet11
    max_peers: 5,
    coinset_fallback_enabled: true,
    ..Default::default()
};

let query = ChiaQuery::new(config).await?;
```

The `ChiaQuery` instance is stored in `ConsensusClient` and shared with the
indexer and all puzzle drivers.

### Methods Used by This Crate

| ChiaQuery Method | Used By | Purpose |
|-----------------|---------|---------|
| `get_coin_records_by_puzzle_hash()` | Indexer (RPC-003) | Find network coin, checkpoint singleton, registration coins |
| `get_coin_record_by_name()` | Indexer (RPC-003) | Look up specific coins by ID |
| `get_puzzle_and_solution()` | Indexer (RPC-003) | Verify lineage, extract curried params |
| `get_blockchain_state()` | Indexer (RPC-003) | Peak height for sync, reorg detection |
| `get_additions_and_removals()` | Indexer (RPC-003) | Block-level coin tracking |
| `get_coin_records_by_parent_ids()` | Puzzle drivers (RPC-002) | Find child coins of singleton spends |

### Methods NOT Used by This Crate

Per API-008, the crate never broadcasts. These methods are available to
the importing project but are not called internally:

- `push_tx()` — caller broadcasts bundles
- `wait_for_confirmation()` — caller monitors inclusion
- `get_fee_estimate()` — caller manages fees (or via dig-l1-wallet)

### Connection Architecture

```
ConsensusClient
    └── ChiaQuery
            ├── Peer Pool (decentralized, up to 5 peers)
            │     ├── 127.0.0.1 (tried first if available)
            │     ├── TRUSTED_FULLNODE env var (priority)
            │     └── DNS-discovered peers
            └── Coinset.org API (fallback when peers fail)
```

### ConsensusClient Integration

```rust
pub struct ConsensusClient {
    config: NetworkConfig,
    query: ChiaQuery,       // blockchain query backend
    indexer: IndexerState,
    proving_key: Option<ProvingKey>,
    // ...
}

impl ConsensusClient {
    pub async fn new(
        config: NetworkConfig,
        query_config: ChiaQueryConfig,
    ) -> ConsensusResult<Self> {
        let query = ChiaQuery::new(query_config).await
            .map_err(|e| ConsensusError::RpcError(e.to_string()))?;
        Ok(Self {
            config,
            query,
            indexer: IndexerState::new(),
            proving_key: None,
        })
    }
}
```

### Error Mapping

| ChiaQueryError | ConsensusError |
|---------------|----------------|
| `PeerConnection` | `RpcError("peer connection failed: ...")` |
| `PeerRejection` | `RpcError("peer rejected query: ...")` |
| `CoinsetApiError` | `RpcError("coinset API error: ...")` |
| `AllSourcesFailed` | `RpcError("all sources failed: ...")` |

### String-Based Identifiers

`chia-query` uses `&str` hex strings (e.g., `"0xabcd..."`) for coin IDs and
puzzle hashes, not `Bytes32`. The crate MUST convert between `Bytes32` and
hex strings at the boundary:

```rust
fn bytes32_to_hex(b: &Bytes32) -> String {
    format!("0x{}", hex::encode(b.as_ref()))
}

fn hex_to_bytes32(s: &str) -> ConsensusResult<Bytes32> {
    let s = s.strip_prefix("0x").unwrap_or(s);
    let bytes = hex::decode(s)
        .map_err(|e| ConsensusError::SerializationError(e.to_string()))?;
    Bytes32::try_from(bytes.as_slice())
        .map_err(|_| ConsensusError::SerializationError("invalid Bytes32 length".into()))
}
```
