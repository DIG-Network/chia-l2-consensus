# SMT-005 — Cross-Implementation Consistency

> **Authoritative requirement:** [SMT-005](../NORMATIVE.md#SMT-005)
> **Verification:** [VERIFICATION.md](../VERIFICATION.md)
> **Tracking:** [TRACKING.yaml](../TRACKING.yaml)
> **CHIP reference:** [chip-groth16-l2-consensus.md](../../../../resources/chip-groth16-l2-consensus.md) — Sparse Merkle Tree

## Summary

The Rust implementation (off-chain proof generation) and Chialisp implementation (on-chain verification) MUST produce identical results. Any divergence causes proofs to fail silently. CI tests must verify this property continuously.

## Specification

### What Must Match

| Component | Rust | Chialisp |
|-----------|------|----------|
| Slot computation | `compute_slot()` | `compute_slot` |
| Active leaf hash | `sha256(pubkey)` | `(sha256 pubkey)` |
| Empty leaf hash | `EMPTY_LEAF_HASH` | `EMPTY_LEAF_HASH` |
| Parent hash | `sha256(left \|\| right)` | `(sha256 left right)` |
| Sibling ordering | Left-first | Left-first |
| Proof structure | `[siblings[0..32]]` | `(siblings ...)` |

### Common Divergence Sources

1. **Sibling ordering:** Left/right swap
2. **Index bit interpretation:** Big-endian vs little-endian
3. **Empty hash constant:** Computed differently
4. **Concatenation order:** In parent hash
5. **Integer encoding:** For slot computation

### CI Test Strategy

```rust
#[test]
fn test_cross_implementation_consistency() {
    // Generate test cases
    let test_pubkeys = generate_random_pubkeys(100);

    // Build tree in Rust
    let mut rust_tree = SparseMerkleTree::new(32);
    for pk in &test_pubkeys {
        rust_tree.insert(pk);
    }
    let rust_root = rust_tree.root();

    // Build tree in Chialisp (via clvm_rs)
    let chialisp_root = run_chialisp_tree_builder(&test_pubkeys);

    // Must match exactly
    assert_eq!(rust_root, chialisp_root, "Root mismatch!");

    // Verify proofs work cross-impl
    for pk in &test_pubkeys {
        let rust_proof = rust_tree.prove(pk);

        // Verify Rust proof in Chialisp
        let chialisp_result = run_chialisp_verifier(
            &sha256(pk),
            &rust_proof,
            &chialisp_root,
        );
        assert!(chialisp_result, "Rust proof failed in Chialisp");
    }
}
```

### Required Test Cases

1. **Empty tree:** Both compute same empty root
2. **Single validator:** Identical root and proof
3. **Two validators:** Both orderings verified
4. **Max validators:** Stress test with many entries
5. **Random validators:** Fuzz testing
6. **Boundary slots:** Slots 0, 2^32-1, etc.

### Failure Symptoms

If implementations diverge:
- Proofs generated off-chain fail on-chain
- No error message — just verification failure
- Checkpoint spends rejected
- Membership queries fail

### Canonical Specification

When in doubt, the CHIP specification is authoritative:
- Sibling order: Left always first
- Index bits: Lowest bit = deepest level
- Hash: SHA-256, standard padding

### Recovery from Divergence

If divergence is discovered:
1. Identify which implementation is wrong
2. Fix to match canonical spec
3. Regenerate all proofs
4. Add regression test
