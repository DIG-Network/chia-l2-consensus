# Requirements Schema

This document defines the data model and conventions for all requirements in the
chia-l2-consensus project.

---

## Three-Document Pattern

Each domain has exactly three files in `docs/requirements/domains/{domain}/`:

| File | Purpose |
|------|---------|
| `NORMATIVE.md` | Authoritative requirement statements with MUST/SHOULD/MAY keywords |
| `VERIFICATION.md` | QA approach and verification status per requirement |
| `TRACKING.yaml` | Machine-readable status, test references, and implementation notes |

Each requirement also has a dedicated specification file in
`docs/design/requirements/{domain}/{PREFIX-NNN}.md`.

---

## Requirement ID Format

**Pattern:** `{PREFIX}-{NNN}`

- **PREFIX**: 3-4 letter domain identifier (uppercase)
- **NNN**: Zero-padded numeric ID starting at 001

| Domain | Directory | Prefix | Description |
|--------|-----------|--------|-------------|
| Network Coin | `network_coin/` | `NET` | Network coin singleton puzzle |
| Registration Coin | `registration_coin/` | `REG` | Validator registration and collateral |
| Checkpoint Singleton | `checkpoint/` | `CHK` | Checkpoint state and verification |
| Groth16 Circuit | `circuit/` | `CIR` | ZK circuit constraints and proof generation |
| Sparse Merkle Tree | `smt/` | `SMT` | Merkle tree structure and proofs |
| Wire Format | `wire/` | `WIRE` | Message formats and serialization |
| Indexer | `indexer/` | `IDX` | Off-chain state tracking |
| Security | `security/` | `SEC` | Security assumptions and properties |
| Deployment | `deployment/` | `DEP` | Deployment and setup procedures |
| Validator | `validator/` | `VAL` | Validator onboarding and operations |
| Withdraw Delay | `withdraw_delay/` | `WDC` | Time-locked collateral release after validator exit |
| Crate API | `crate_api/` | `API` | Public API conformance to spec-consensus-crate.md |

**Immutability:** Requirement IDs are permanent. Deprecate requirements rather
than renumbering.

## Requirement Keywords

Per RFC 2119:

| Keyword | Meaning | Impact |
|---------|---------|--------|
| **MUST** | Absolute requirement | Blocks "done" status if not met |
| **MUST NOT** | Absolute prohibition | Blocks "done" status if violated |
| **SHOULD** | Expected behavior; may be deferred with rationale | Phase 2+ polish items |
| **SHOULD NOT** | Discouraged behavior | Phase 2+ polish items |
| **MAY** | Optional, nice-to-have | Stretch goals |

---

## Status Values

| Status | Description |
|--------|-------------|
| `gap` | Not implemented |
| `partial` | Implementation in progress or incomplete |
| `implemented` | Code complete, awaiting verification |
| `verified` | Implemented and verified per VERIFICATION.md |
| `deferred` | Explicitly postponed with rationale |

## TRACKING.yaml Item Schema

```yaml
- id: PREFIX-NNN           # Requirement ID (required)
  section: "Section Name"  # Logical grouping within domain (required)
  summary: "Brief title"   # Human-readable description (required)
  status: gap              # One of: gap, partial, implemented, verified, deferred
  spec_ref: "docs/design/requirements/{domain}/{PREFIX-NNN}.md"
  tests: []                # Array of test names or ["manual"]
  notes: ""                # Implementation notes, blockers, or evidence
```

## Compiled Puzzle Artifacts

On-chain puzzles are authored in Rue (`puzzles/*.rue`) and compiled to two
artifacts that the Rust driver code loads at compile time:

| Artifact | Generated by | Contents | Loaded via |
|----------|-------------|----------|------------|
| `puzzles/compiled/{name}.hex` | `rue build -x` | Serialized CLVM bytecode | `include_str!` in `src/puzzles/{name}.rs` |
| `puzzles/compiled/{name}.hash` | `rue build --hash` | Tree hash (0x-prefixed, 32 bytes) | `include_str!` in `src/puzzles/{name}.rs` |

The `.hex` file is the uncurried puzzle bytecode. Currying happens at runtime in
the Rust driver by prepending parameters.

The `.hash` file is the `MOD_HASH` — the tree hash of the uncurried puzzle. It
is used by:
- The **network coin** to compute the registration coin's curried puzzle hash (NET-003)
- The **indexer** for lineage verification (IDX-002)
- The **deployment runbook** to derive puzzle hashes before on-chain deployment (DEP-002)

### Rebuild Commands

```bash
rue build -x puzzles/{name}.rue > puzzles/compiled/{name}.hex
rue build --hash puzzles/{name}.rue > puzzles/compiled/{name}.hash
```

### Puzzle Inventory

| Puzzle | Source | Hex | Hash | Driver |
|--------|--------|-----|------|--------|
| Network Coin | `puzzles/network_coin_inner.rue` | `compiled/network_coin_inner.hex` | `compiled/network_coin_inner.hash` | `src/puzzles/network_coin.rs` |
| Registration Coin | `puzzles/registration_coin.rue` | `compiled/registration_coin.hex` | `compiled/registration_coin.hash` | `src/puzzles/registration_coin.rs` |
| Checkpoint Singleton | `puzzles/checkpoint_inner.rue` | `compiled/checkpoint_inner.hex` | `compiled/checkpoint_inner.hash` | `src/puzzles/checkpoint.rs` |

## Hard Testing Requirements for On-Chain Puzzles

All puzzle-domain requirements (NET, REG, CHK) MUST meet these testing
standards. Source-inspection-only tests are **not acceptable** for verifying
puzzle behavior.

### 1. CLVM Execution Tests (MUST)

Every puzzle VV test MUST actually execute the compiled CLVM bytecode:

1. **Deserialize** the `.hex` artifact into a `NodePtr` via `clvmr`
2. **Curry** the puzzle with test parameters using `CurriedProgram`
3. **Run** the curried puzzle with a test solution via `run_program()`
4. **Parse** the output condition list
5. **Assert** exact condition opcodes, hashes, and amounts

Source-string inspection (e.g., checking `src.contains("AggSigMe")`) is
permitted only as a supplement, never as the sole verification.

### 2. chia-wallet-sdk Simulator Tests (MUST)

All puzzle spend paths MUST be tested using the `chia-sdk-test::Simulator`:

1. **Create** a `Simulator` instance
2. **Mint** test coins with known puzzle hashes
3. **Build** spend bundles with `SpendContext`
4. **Submit** via `sim.spend_coins()` or `sim.new_transaction()`
5. **Verify** resulting coin states: created coins, spent coins, amounts, puzzle hashes
6. **Test failure cases**: invalid signatures, wrong announcements, insufficient funds

The simulator runs full consensus validation including CLVM execution, signature
aggregation, and announcement matching.

### 3. Permutation Testing (MUST)

Tests MUST cover all permutations of:

| Dimension | Examples |
|-----------|---------|
| **Curried parameters** | Different pubkeys, different checkpoint IDs |
| **Solution values** | Valid/invalid epochs, destinations, amounts |
| **Spend paths** | Each spend path of multi-path puzzles (checkpoint vs membership query) |
| **Success cases** | Valid inputs produce expected conditions |
| **Failure cases** | Invalid inputs cause spend rejection |
| **Edge cases** | Zero amounts, max values, empty conditions list |
| **Cross-implementation** | Rust-computed hashes match CLVM-computed hashes |

### 4. Required Test Infrastructure

```toml
# Cargo.toml [dev-dependencies]
chia-sdk-test = "0.18"     # Simulator, test keys, sign_transaction
```

```rust
// Test utilities available:
use chia_sdk_test::{Simulator, test_secret_key};
use chia_sdk_driver::SpendContext;
use clvmr::{Allocator, ChiaDialect, run_program, serde::node_from_bytes};
use clvm_traits::{ToClvm, FromClvm};
use clvm_utils::CurriedProgram;
```

### 5. Test File Structure for Puzzle Domains

```rust
//! REQUIREMENT: REG-001 — Registration Coin Puzzle Structure
//!
//! CLVM execution tests: deserialize hex, curry, run, parse conditions.
//! Simulator tests: full spend bundle lifecycle.

// ── CLVM execution tests ───────────────────────────────────────────
// Deserialize .hex, curry with params, run with solution, check conditions

// ── Simulator tests ────────────────────────────────────────────────
// Create coins, build spends, submit to simulator, verify state

// ── Cross-implementation tests ─────────────────────────────────────
// Rust hash computation matches CLVM hash for same inputs

// ── Failure case tests ─────────────────────────────────────────────
// Invalid inputs rejected by CLVM or simulator
```
